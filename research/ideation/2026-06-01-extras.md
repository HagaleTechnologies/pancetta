## X1 — Hardware-conditioned PSK-radio noise blanker fed from CAT

CATEGORY: hardware-rx-side / cognitive-radio

### Mechanism
Modern HF rigs (the FTdx10 is one) expose noise blanker / noise
reduction / contour DSP knobs via CAT (hamlib). Today pancetta uses
the receiver as a dumb audio source. The proposed mechanism: feed
the decoder's **per-pass residual energy histogram** back to the rig
through CAT to adapt NB width, NR depth, and IF passband on a
per-band basis. When pancetta sees an impulsive QRM signature in
residuals (broad spectral spread, narrow time peaks), it commands NB
narrower; when it sees broadband white-ish noise, it widens IF
passband to recover SNR.

The decoder closes the loop: each adjusted-rig slot is graded by
total decode count and mean SNR. A small RL bandit (UCB or
Thompson) over the discrete CAT knob grid converges in ~50 slots
per band. Key novelty: pancetta-as-the-rig-tuner, not the operator.

The pancetta-hamlib crate already has CAT plumbing; the bandit is
~100 LOC; the residual histogram is the existing per-pass spectrogram
delta. Plumbing exists.

### Category
Hardware / RX-side (with a feedback-loop cognitive-radio twist).

### Defensible prior
FTdx10 manual documents NB1/NB2/CONTOUR/WIDTH each affect HF weak-signal
copy; WSJT-X community wisdom is "find the rig settings for your noise
floor". Automating the search closes the operator-knowledge gap.
hamlib `rigctl` supports `L NB`, `L NR`, `B`, `W` commands on the FTdx10.

### Assumption challenged
The receiver is fixed and the decoder must work with whatever audio
shows up. (False — the rig is programmable; pancetta can drive it.)

### Kill-switch sketch
2-band, 1-hour A/B test on live air: decoder-driven rig settings vs.
operator's hand-tuned settings (recorded baseline). PROCEED if
decode count ≥ baseline AND mean SNR within 0.5 dB on at least one
band. KILL if any band loses ≥ 5% decode count vs hand-tuned.

### Effort estimate
3-4 sessions (Session 1 = histogram + bandit primitive; Session 2 =
CAT integration via hamlib; Session 3 = live A/B; Session 4 =
production wiring).

### Headline risk
Rig-state contention with operator: if K5ARH manually tweaks NB mid-QSO,
the bandit's state model goes stale. Mitigation: bandit pauses when
CAT readback shows operator-initiated changes.

## X2 — TX-time micro-jitter to escape repeated collisions

CATEGORY: tx-side wins / cognitive-radio

### Mechanism
FT8 protocol slots are nominally 15.0s, but every TX-er has DT
variance (0.1-0.5s typical). If two stations on the same frequency
have correlated DT (e.g., both auto-disciplined to GPS), they'll
collide cycle after cycle. The proposed mechanism: pancetta adds a
small **per-cycle TX-time jitter** (±50-200 ms uniform) to escape
phase-locked collisions with another auto-TX-er sharing its
frequency.

The jitter applies only when the autonomous responder detects a
repeated collision (same target callsign failing to copy us at the
same dt offset over 2+ slots). Standard slot timing for new QSOs;
randomized timing only when collision-recovery is needed.

Critically, this is *anti-coordinated* with other auto-stations:
WSJT-X and similar all aim for 0.0s exactly, so any small jitter
in our favor de-correlates us from their pile-up.

### Category
TX-side wins (novel — TX-jitter as a contention-avoidance primitive).

### Defensible prior
CSMA backoff in 802.11 uses random delay to break collisions; same
principle. FT8 spec permits DT up to ±2.5s before the receiver
fails to lock. Operators routinely tolerate ±0.3s in real bands.

### Assumption challenged
"Closer to 0.0 DT is always better." False when another stable
TX-er has *also* hit 0.0; collision becomes phase-stationary.

### Kill-switch sketch
Loopback simulator with two simulated TX-ers on same freq+target.
Measure QSO completion rate with vs without jitter over 100 simulated
slot pairs. PROCEED if jitter completes ≥ 20% more QSOs. KILL if
non-collision-case QSO rate drops by > 2%.

### Effort estimate
1-2 sessions (jitter primitive + state-machine integration).

### Headline risk
Operator perception of "off-time" TX. Mitigation: jitter only on
detected collision; log every jitter event so operator can see why.

## X3 — SDR I/Q tap upgrade path with phase-coherent A/B decode

CATEGORY: hardware / rx-side / diversity-adjacent (but NOT antenna diversity)

### Mechanism
Today pancetta consumes 16-bit PCM at 12 kHz from the rig's USB
audio. Modern SDRs (RSPdx, KiwiSDR, even RTL-SDR with care) provide
full I/Q at hundreds of kHz. Proposed mechanism: add a *parallel*
SDR I/Q ingest path (`pancetta-audio` extended with an I/Q source
trait), then run **both** the rig's PCM and the SDR's I/Q through
decode in parallel for the same slot. Bit-exact compare on common
decodes; report SDR-only / rig-only as evidence of receiver-side
gaps.

The novelty over hb-028 (cross-decoder ensemble): same decoder, two
RX chains. Isolates RX hardware contribution from decoder
contribution. Once we know which chain wins by how much per band,
we can recommend SDR-upgrade ROI to other operators.

Phase-coherent fusion (FT8 is single-tone-per-symbol — a coherent
matched filter combining two RX chains at known truth coords could
yield ~3 dB) is the upgrade path beyond plain A/B.

### Category
Hardware-rx-side (with phase-coherent fusion as the eventual reach).

### Defensible prior
KiwiSDR + WSJT-X is a well-known offline workflow; the operational
gap is no live A/B harness. Phase-coherent receive diversity is
standard in HF: 3 dB array gain for 2 chains, ~5 dB for noise
decorrelation cases.

### Assumption challenged
"The rig is the best RX we'll get." Untested for HF weak-signal.

### Kill-switch sketch
1-week passive A/B: SDR I/Q logged alongside rig PCM on same antenna.
Decode both in batch; report SDR-only decodes, rig-only decodes,
common decodes. PROCEED to coherent-fusion phase if SDR delivers
≥ 5% more unique decodes on hard-200-style real corpus.

### Effort estimate
4-6 sessions (Session 1-2 = I/Q ingest trait; Session 3 = SDR
downconvert to 12 kHz; Session 4 = parallel decode pipeline;
Session 5-6 = coherent fusion).

### Headline risk
Hardware investment for the operator. Initial A/B can be done with
a $30 RTL-SDR before committing to better gear; defensible spend.

## X4 — Decoder warm-up via off-air pre-load of last-cycle FFT plans + Costas templates

CATEGORY: cold-start / dev-infra

### Mechanism
On launch, pancetta cold-starts: first slot decode is measurably
slower (FFT plan allocation, Costas template instantiation, OSD
lookup-table priming). For an autonomous station rebooted after
power loss or update, the first slot can be lost entirely if
warm-up exceeds 15s.

Proposed mechanism: serialize the **hot decoder state** to disk on
clean shutdown (FFT plan twiddle factors, Costas symbol templates
for sample rate, OSD HRT-permutation buffers, BP message-buffer
allocations). On startup, mmap-restore in < 100 ms. The wake-up
QSL slot is decoded normally.

Additionally: pre-warm the decoder thread with a no-op slot of
synthetic silence to JIT the rayon thread pool before the first
real slot arrives.

### Category
Operational hygiene / dev-infra / cold-start latency.

### Defensible prior
JIT'd code paths show 2-10× latency for first call (well-known
hot/cold split). FFTW + rustfft both have plan-caching APIs. The
literature on cold-start in serverless (firecracker) makes the
same trade-off.

### Assumption challenged
"Pancetta is always already warm." False on every boot, update,
and recovery-from-crash.

### Kill-switch sketch
Measure first-slot decode latency cold vs warm in instrumented
runs. PROCEED if cold-start exceeds 10s on the operator's MiniPC
in instrumented tests; SHELVE if cold-start is consistently < 5s
(no real-world cost).

### Effort estimate
1-2 sessions.

### Headline risk
Stale serialized state on decoder code changes. Mitigation: version
tag on serialized blob; mismatch → cold start as today.

## X5 — WSJT-X plug-in adapter (pancetta-as-decoder for other front-ends)

CATEGORY: integration / community-lever

### Mechanism
WSJT-X uses `jt9` (binary) as its decoder. Many community front-ends
(JTDX, WSJT-Z, MSHV) ship variants. Proposed mechanism: ship a
`pancetta-decoder` CLI binary that takes a WAV on stdin and writes
WSJT-X-format decode lines on stdout — drop-in for `jt9 --ft8`. Now
any operator running WSJT-X can A/B pancetta vs jt9 by changing
one PATH symlink.

The novelty: pancetta-research already has the decoder; the CLI
wrapper is ~200 LOC. The headline result becomes "pancetta decodes
X% more than jt9 on YOUR shack", measured by the user's own ear.
That is the strongest possible community lever; if pancetta wins
on diverse user shacks, momentum follows.

A secondary benefit: WSJT-X UI captures operator-validated QSOs,
which is the log-as-truth feedback loop we've been wanting (see X9).

### Category
Integration / community / WSJT-X plugin.

### Defensible prior
jt9's I/O format is documented (WSJT-X source, GPL); the CLI surface
is stable. JTDX is literally a fork that did exactly this with a
different decoder. Precedent exists.

### Assumption challenged
Pancetta must be a complete station to be useful. False — it can be
an upgrade to existing stations.

### Kill-switch sketch
Build CLI; run on 3 community-volunteer shacks for 1 weekend each.
PROCEED to community release if pancetta wins on ≥ 2 of 3 by
≥ 5% decode count.

### Effort estimate
3-4 sessions (Session 1 = CLI scaffold + format adapter; Session
2 = WSJT-X integration test; Session 3-4 = volunteer rollout).

### Headline risk
Compatibility tail (WSJT-X version diversity, line-format edge
cases). Mitigation: pin to current WSJT-X stable; bug-bash before
rollout.

## X7 — Adaptive notch from chirp-and-carrier detector for HF QRM

CATEGORY: DSP-level / hardware-rx-side

### Mechanism
The hard-200 corpus has WAVs where a single carrier or chirped
QRM (loose dimmer, switch-mode power supply, OTH radar sweep)
sits in the FT8 band and corrupts adjacent FFT bins. Today
pancetta has no QRM-specific defense.

Proposed mechanism: a real-time **carrier-and-chirp detector**
on the input audio (CFAR detector with chirp-rate matched filter)
identifies persistent narrow-band offenders. Pancetta then
applies an **adaptive notch filter** at those frequencies before
the FFT, sacrificing the decoder's ability to find truths *under*
the notch (acceptable — already failing) to clean up the
decoder's ability to find truths *adjacent* to the notch (the
real win).

Synergistic with X1 (CAT-driven rig NB) but acts at the audio
DSP level, not the rig level — needed when the rig's notch is
already saturated or absent on that band.

### Category
DSP-level (with overlap to RX hygiene).

### Defensible prior
Adaptive notch is standard in HF (LMS filters, frequency-domain
nulling). Joe Taylor's WSPRdaemon includes a carrier detector for
exactly this. CFAR detectors are standard radar primitives.

### Assumption challenged
"FFT bin leakage from QRM is something we tolerate." Active
suppression is feasible at audio-rate.

### Kill-switch sketch
Synthetic corpus of FT8-with-QRM (CW carrier injected at random
freq in band) over 100 slots. Measure decode count with vs without
notch. PROCEED if notch recovers ≥ 5% decodes lost to QRM. KILL if
notch suppresses ≥ 1 truth on QRM-free synth slots (false-notch
case).

### Effort estimate
2-3 sessions.

### Headline risk
Notch overshoots and kills truths adjacent to the QRM. Mitigation:
narrow notch (~30 Hz wide), CFAR threshold tuned for high-confidence
detection only.

## X8 — Log-as-ground-truth: operator-completed QSOs validate decodes retroactively

CATEGORY: operational hygiene / human-loop-adjacent

### Mechanism
Every QSO K5ARH completes through pancetta produces ADIF entries with
verified callsign, grid, RST. Each completion implies that *some
prior decode* of that callsign was correct. Proposed mechanism: a
**retroactive labeler** that consumes the operator's ADIF log,
matches each completed QSO to the candidate decodes in the
hours leading up to it, and labels those decodes as **operator-
validated true positives**.

This grows a corpus of *real-band* truth labels without any manual
labeling effort, at the cost of one ADIF-to-decode join. Once large
enough, this corpus becomes:
1. A precision benchmark complementary to hard-200 (which is
   synthetic-truth in real-band noise).
2. Training data for any learned-component hypothesis (DIA-OSD,
   neural soft-demod) that's been bottlenecked on corpus size.
3. A regression detector — if a decoder change drops operator-
   validated decodes, we know immediately, even without an
   eval suite run.

Differs from hard-200 (manually truth-labeled) in that this corpus
auto-grows with every QSO.

### Category
Operational hygiene / corpus-adjacent (but mechanism is the join,
not the corpus itself — distinct from the corpus category).

### Defensible prior
ADIF is the universal QSO log format and pancetta already writes it.
WSJT-X's "Show DXCC" feature does a similar partial-join in the
opposite direction. The ground-truth label structure is conservative
(implies only "this callsign was on-air around this time").

### Assumption challenged
"Truth labels for real-band decodes are expensive." False — the
operator has been producing them since Phase 4.

### Kill-switch sketch
1-week passive run: count operator-validated true positives in the
ADIF-decode join. PROCEED if ≥ 50 labels/week (enough to grow a
~2000-label corpus over a year). SHELVE if < 10/week (insufficient
yield).

### Effort estimate
1-2 sessions (Session 1 = ADIF-to-decode-log join primitive;
Session 2 = label corpus packaging into pancetta-research).

### Headline risk
Time-window ambiguity (multiple decodes of same callsign in window).
Mitigation: label all matches in the window; downstream uses can
filter.

## X9 — Differential decode: same WAV processed twice with permuted candidate order

CATEGORY: differential analysis / dev-infra

### Mechanism
Pancetta decodes candidates in (approximately) descending sync-score
order, and earlier decodes coherent-subtract from the residual seen
by later decodes (hb-079 multipass). The hidden assumption: the
greedy-order is decoupled from the final decode set.

Proposed mechanism: run **two parallel decode passes** on the same
WAV with **different candidate orderings** (e.g., descending sync
score vs ascending time-bin order vs random seed). Take the
**union** of CRC-passing decodes from both. Where the union exceeds
either single pass, the per-WAV delta tells us:
1. How much order-coupling exists in multipass (diagnostic).
2. Whether a multi-order ensemble is a recall lever (operational).

Different from hb-028 (cross-decoder ensemble — different decoders);
this is cross-order, *same decoder*. Distinct from hb-079's
multipass (which is single-order iterative).

The novelty: orderings are cheap (~2× decode time), and the gap
between best-order and worst-order quantifies the
multipass-greediness penalty.

### Category
Differential analysis / dev-infra.

### Defensible prior
Greedy algorithms are well-known to suffer order-dependence; iterative
subtraction (hb-079) is a greedy algorithm. The fix is well-trodden:
multi-order union. Tools like LDPC decoder schedules already do
something similar with permuted message orders.

### Assumption challenged
"Sync-score-descending order is optimal." Untested.

### Kill-switch sketch
hard-200 single-pass diagnostic over 10 random candidate orderings.
Measure per-WAV union vs max single-pass. PROCEED if median union
exceeds max single-pass by ≥ 1 decode per WAV.

### Effort estimate
1 session (diagnostic; if promising, production wiring is hb-X9b).

### Headline risk
2× compute cost for marginal gain. Mitigation: only run multi-order
on WAVs where first pass found < N decodes (suggests a hard WAV).

## X10 — Anomaly detection on residuals: ML-trained "weird residual" tagger

CATEGORY: ML/learned (residual-quality, distinct from hb-094/095 angles)

### Mechanism
Post-multipass residuals (hb-079 saturated) contain a mix of
noise, interferer leakage, and *occasionally* still-buried weak
truths. Today we run decode on all residual positions equally.

Proposed mechanism: train a **small CNN anomaly detector**
(MobileNet-style, < 100k params) on residual spectrogram patches
labeled with "decoded truth here" vs "no truth here" using the
synth corpus where we know ground truth. The detector outputs a
**per-position anomaly score**; we decode only the top-K anomalous
positions.

Distinct from foundation-models category (this is a small, focused
binary classifier — not a foundation model). Distinct from hb-094
(residual denoising autoencoder) which acts on pixels to recover
SNR. Distinct from hb-095 (neural soft-demod) which replaces LLR
extraction. Distinct from hb-064 (DIA-OSD) which acts post-BP.

This is a **pre-decode position gate**, learned. Complementary to
hb-093 (analytical residual SNR gate) — could even ensemble.

### Category
Anomaly detection / ML-learned (residual-position gate).

### Defensible prior
Anomaly detection on spectrograms is a solved problem class (sound
event detection, machine-condition monitoring). MobileNet-V3-small
runs in microseconds on the operator's MiniPC. Synth corpus growth
plan (hb-100) provides training data; X8 grows it further with
real-band labels.

### Assumption challenged
"Decode every position that passes sync." Wasted LDPC work on noise
positions is the bottleneck.

### Kill-switch sketch
Train classifier on hb-100 synth corpus. Measure precision/recall
on held-out synth + hard-200 top-20 diagnostic. PROCEED if
precision ≥ 0.90 at recall ≥ 0.95 (i.e., we can skip ≥ 10% of
positions with < 5% truth loss). KILL if precision-recall AUC
< 0.75 on real-band hard-200.

### Effort estimate
3-4 sessions (Session 1 = data prep; Session 2 = model training;
Session 3 = inference primitive; Session 4 = production wiring).

### Headline risk
Distribution drift (training-time synth doesn't match production
real-band). Mitigation: continuous fine-tune from X8 operator-
validated labels.

## X11 — Wiener filter on residual spectrogram (SNR-optimal pre-decode cleanup)

CATEGORY: DSP-level

### Mechanism
The optimal MMSE filter for a known signal in additive noise of
known PSD is the Wiener filter. After hb-079 multipass saturation,
the residual spectrogram has a *measurable* noise floor (per-bin
mean and variance over no-truth positions). The proposed mechanism:
compute the per-bin noise PSD from non-Costas-aligned positions,
then apply a Wiener filter to the residual before the final decode
pass.

Synergistic with hb-079 (multipass) and hb-090 (matched filter):
matched filter is *signal-shape* optimal; Wiener is *noise-spectrum*
optimal. They combine multiplicatively (Wiener-pre then matched-
filter-extract).

Different from a denoising autoencoder (X10's territory) — Wiener is
analytical, requires no training, and has provable optimality under
its assumptions. Different from notch (X7) — Wiener attenuates
based on per-bin SNR continuously, not at sharp frequency cuts.

### Category
DSP-level.

### Defensible prior
Wiener filter optimality (Wiener 1949, well-cited in any DSP
textbook). HF radio applications: WSPR uses adaptive equalizers
with similar derivation. The PSD estimator is the existing
`par_estimate_snr_spectrogram` re-aimed at no-truth positions.

### Assumption challenged
"The residual is already as clean as analytical DSP can make it."
Untested with proper Wiener cleanup.

### Kill-switch sketch
hard-200 top-20 diagnostic — measure per-position residual SNR
before and after Wiener filter, on truth-coords for missed truths.
PROCEED if median post-Wiener SNR improves ≥ 1 dB AND additional
decode count on follow-up pass ≥ 5 across 20 WAVs.

### Effort estimate
1-2 sessions.

### Headline risk
Stationarity assumption (Wiener assumes WSS noise). HF noise is
non-stationary. Mitigation: short-time Wiener (per-slot windowed
PSD); the residual is already slot-local.

## X12 — GPU/SIMD micro-architecture win: AVX-512 LDPC BP on operator's MiniPC

CATEGORY: micro-architecture / performance

### Mechanism
Today pancetta's BP iterates in scalar Rust over a small set of
parallel decode threads (rayon). The operator's MiniPC has AVX-512
(modern Intel/AMD). Proposed mechanism: rewrite the BP inner loop
(min-sum on Tanner graph edges) using **packed SIMD** — AVX-512
yields 16-way f32 parallelism per instruction, ~4-8× speedup over
scalar. With 100 BP iterations × 600 codewords × 174 bits, the
inner loop dominates decoder runtime.

Speedup is *not* the headline goal — the headline is that elapsed
budget for new mechanisms expands. Today hb-093 / hb-094 / hb-090
are partially elapsed-budget-gated; a 3× BP speedup gives 3× more
elapsed headroom for new mechanisms in the same wall-clock budget.

Alternative target: GPU offload (vulkan / wgpu / CUDA) for batch
LDPC decode. WSJT-X-Improved has experimented with this. Wild-card
extension: the same GPU can run a residual-anomaly CNN (X10) for
free in idle slot time.

### Category
Micro-architecture / performance / elapsed-budget enabler.

### Defensible prior
SIMD LDPC implementations published (GPU: open-source 5G NR
decoders; CPU: NIST's BIKE/HQC reference code uses AVX2 for LDPC-
adjacent ops). Speedups of 4-10× are routine for inner-loop SIMD.
Rust `std::simd` is stabilizing.

### Assumption challenged
"BP is fast enough." False given elapsed-budget pressure on new
mechanisms.

### Kill-switch sketch
Benchmark scalar BP vs SIMD BP on the operator's MiniPC. PROCEED
if ≥ 2× speedup on full-decode workload. KILL if speedup < 1.3×
or bit-exactness fails on any hard-200 WAV.

### Effort estimate
3-5 sessions (Session 1 = inner-loop profiling; Session 2-3 =
SIMD rewrite; Session 4 = bit-exact verification; Session 5 =
production wiring).

### Headline risk
SIMD bug introduces non-bit-exactness; bit-exactness with ft8_lib
is a core invariant. Mitigation: bit-exact test suite (~295 tests)
must pass before merge; SIMD path opt-in until validated.

## X13 — Streaming continuous-spectrogram decoder (no slot batching)

CATEGORY: streaming / architecture (cross-cutting; NOT in the
architectural category which focuses on decoder-internal refactor)

### Mechanism
Today pancetta decodes each 15s slot as a batch: collect 15s of
audio, FFT it, decode candidates, repeat. Proposed mechanism:
a **rolling-buffer continuous decoder** that incrementally updates
a sliding spectrogram and runs partial decode passes every ~1s.

Benefits:
1. The decoder can start finding truths *before* the slot ends —
   pairs with hb-091 (a8 early-decode) for true partial-slot
   decode, not just-in-time.
2. Slot-edge truncation (DT > 1.5s) becomes a soft constraint
   instead of hard truncation — the decoder sees the truth even
   when it crosses slot boundaries.
3. Slot-alignment errors (PC clock skew) become recoverable —
   the decoder finds the slot edges itself from the data.

Pancetta's current architecture batches by design (the QSO state
machine assumes slot-aligned decode events). Implementation can
preserve this by emitting decodes only at slot boundaries while
the *internal* decoder runs continuously.

Differs from X12 (which speeds up batched decode) — this changes
the decode granularity.

### Category
Streaming architecture / cross-cutting (latency + recall).

### Defensible prior
SDR-style streaming decoders are the rule in WSPR (WSPRdaemon
streams continuously). FT8 was originally designed for batch
because of WSJT-X's PC-clock-aligned slot model, but the protocol
permits continuous decoding.

### Assumption challenged
"FT8 is a slot-batched protocol." The protocol structure is slot-
batched; the decoder need not be.

### Kill-switch sketch
Build streaming primitive on a 1-hour passive log. Compare per-
decode CRC-pass rate with batch vs streaming. PROCEED if streaming
finds ≥ 5% additional decodes (slot-edge cases). KILL if streaming
introduces any FP not present in batch.

### Effort estimate
4-6 sessions (architectural change; significant test surface).

### Headline risk
Architectural disruption to coordinator's slot-aligned QSO state
machine. Mitigation: streaming runs *parallel* to batched until
proven; coordinator continues consuming slot-aligned decode events.

## X14 — Per-band propagation prior from VOACAP / ITU-R for AP weight tuning

CATEGORY: propagation-aware (decoder-side; distinct from
human-loop or operational priority scoring)

### Mechanism
At the decoder level, AP (a-priori) message-template injection
benefits from priors about *what messages are plausible* on this
band right now. Proposed mechanism: integrate a **VOACAP-style
propagation prediction** that, given UTC time + band + grid square,
returns a per-DXCC reachability probability. Multiply that prior
into AP candidate ranking — boost CQ-from-likely-DXCC templates,
de-rank CQ-from-unlikely-DXCC templates.

When 20m is dead to Europe at 0200Z local, AP shouldn't waste cycles
testing OE/HA/UA callsign templates. When 20m is wide open to JA at
