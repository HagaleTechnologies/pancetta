## Framing

The corpus is the instrument. Every graduated hypothesis (27 of them
across 5 days) is measured against hard-200 + hard-1000 + synth-clean
+ synth-doppler + wild-50/100. The instrument shapes which decoder
behaviors we reward. Tonight's structural-ceiling closures (V2/V3
soft-cancellation shelved, hb-088 marginal, hb-068 b-0.25 +7 hard-200
only) all hit walls **defined by this corpus**. If the corpus
mismeasures the recall opportunity, the iter cycle is fitting to the
wrong target.

This file enumerates 14 alternative corpora that would stress decoder
behaviors hard-200/1000 + synth doesn't. Each entry challenges a
specific assumption about what "good FT8 decoding" means in pancetta's
target deployment.

Existing corpora we are NOT duplicating:
- hard-200 (K5ARH 20m busy-NA, refreshed 2026-05-31)
- hard-1000 (same source, 5× wider sample)
- wild-50/100 (operator-typical, production tracking)
- synth-clean (controlled SNR sweep)
- synth-doppler (rate-controlled drift, currently un-decoded)
- wild-doppler-50 (hb-073 scope, capture pending)

## C1 — Cross-decoder consensus truth (panc ∩ jt9 ∩ jtdx)

**Description.** Build a "high-confidence truth" tier where ground-truth
decodes are only those agreed-on by three independent decoders:
pancetta@HEAD, WSJT-X jt9, and JTDX. The corpus is the same audio set
as hard-200 (or a larger sample); the change is the truth function.
Each WAV gets a `truth_decodes` list = intersection of all three
decoders' message-text sets. Pancetta-only and pancetta-novel decodes
are scored separately as "candidate" decodes that don't count for
recall but DO count for FP-risk auditing.

Collection: zero new audio. Add jtdx as a baseline backend in
`baseline.rs` (already runs jt9 via subprocess; jtdx is a fork with
the same CLI shape). One-time cost.

**What it would reveal.** Today's hard-200 truth is jt9-only. Pancetta
"novel" decodes (currently 1970 on refreshed hard-200) include
genuine TPs jt9 missed AND silent pancetta FPs we can't catch.
Stricter consensus truth would: (1) lower the inflated novel number
to a defensible floor, (2) expose WAVs where jtdx beats jt9 — a
different decoder regime — and (3) give us a "panc-vs-jtdx" gap
analysis the current corpus can't surface.

**Defensible prior.** WSJT-X's release validation uses multi-decoder
agreement informally; arXiv 2402.xxx (academic LDPC papers) routinely
cite "intersection of N decoders" as gold-truth. JT9 + JTDX are
deliberately independent code-bases.

**Assumption challenged.** "jt9 is good enough as truth."

**Kill-switch sketch.** Bootstrap is trivial — same WAVs, add jtdx as
second baseline. If jtdx adds <5% novel decodes vs jt9 on a 20-WAV
pilot, the consensus tier collapses to ~jt9 truth and isn't worth
building. Pilot in 1 hour.

**Effort estimate.** Collection: 0. Curation: 0. Integration: ~4 h
(add `--baseline-engine jtdx` to baseline.rs, schema bump for
consensus column, eval dispatch entry). jtdx install: ~30 min.

**Headline risk.** jtdx is a moving-target fork; its behavior changes
release-to-release. Pin a specific jtdx version + commit it to the
operator's machine to keep the truth function reproducible.

## C2 — Pancetta-self-truth recursive refinement

**Description.** Use pancetta@HEAD's HIGH-CONFIDENCE decodes (CRC pass,
LDPC iters < N, OSD depth ≤ 2, plurality vote across multi-pass)
as ground truth for a "harder" tier. Each WAV's truth set =
pancetta's most-confident decodes; recall measures whether
pancetta@CANDIDATE recovers them under a tighter decode budget
(fewer passes, lower OSD, lower max-candidates).

Collection: zero new audio. Same hard-200/1000 WAVs.

**What it would reveal.** Whether decoder iterations actually improve
the **robust** decode set, or just expand the noisy tail. Today,
"recall +12" could mean +12 genuine signal recoveries OR +12
weak/risky decodes that happen to pass CRC. This tier rewards
stability under budget pressure — the deployment regime that
matters for autonomous QSO completion.

**Defensible prior.** wild_card: false. Bootstrapping classifiers on
their own confident predictions is standard in semi-supervised ML
(noisy-student, self-training). Conformal prediction uses similar
self-calibration.

**Assumption challenged.** "External truth (jt9) is the only honest
truth." Self-truth captures pancetta-specific behavior the external
decoder is blind to (e.g., the V3 subtract-aware sync gains).

**Kill-switch sketch.** Run pancetta on hard-200 with maximum budget
(max_passes=8, osd_depth=6) and tightest config; capture the
"truth" set. Run again with deployment budget (max_passes=3,
osd_depth=2). If recall ratio > 99%, there's no headroom — the
decoder is already saturated under budget. If ratio < 90%, budget
matters and this tier is the right instrument for hb-016 residual-
energy-stop and hb-068 b-scale tuning.

**Effort estimate.** ~6 h: new tier in eval.rs, "high-confidence"
truth generation script (1 LOC change to existing baseline.rs),
manifest generation.

**Headline risk.** Self-truth creates a fixed point: decoder learns
to recover its own decodes, masking real recall gaps. Mitigate by
forbidding this tier from being the primary recall metric — it's a
**stability** metric, not a recall metric. Composite weight should
be near zero; pair with cross-decoder consensus (C1).

## C3 — Synthetic adversarial corpus targeting measured walls

**Description.** Algorithmically generate WAVs that maximize pancetta's
known failure modes. Three sub-families:
- **mutual-masking pairs**: two FT8 signals at ΔF ∈ {3, 6, 12, 25} Hz
  and ΔSNR ∈ {0, +3, +6, +10} dB. Targets hb-086 V2/V3 (joint
  decoding regime).
- **sub-Costas signals**: SNR ∈ [-26, -22] dB where Costas sync
  scores fall below threshold. Targets hb-088 (low-Costas tail).
- **3+ collision stacks**: 3-5 signals overlapping in a ±30 Hz band.
  Targets joint-pair-retry's known weakness past N=2.

All synthesized with the existing `gen-synth` infrastructure; new
config types. Each sub-family ~50 WAVs.

**What it would reveal.** Exactly which hypotheses can move the needle
on the wall they were designed for. Today, V2 was shelved at 0%
marginal-neighbor on old hard-200; refreshed corpus showed more
marginal cases but still organic. Adversarial generates the wall
on demand.

**Defensible prior.** Adversarial ML — generate inputs that maximize
loss. WSJT-X test suite includes contrived test cases for
collision edge-cases (`wsjtx/tests/...`). MAP-equivalent decoder
papers (Yedidia et al.) routinely use targeted synthetic stress
cases.

**Assumption challenged.** "Real-world corpus naturally exercises all
decoder failure modes." It doesn't — real audio's failure modes
follow a long-tail distribution; adversarial corpus puts the tail
in the head.

**Kill-switch sketch.** Build the mutual-masking pair generator
(<1 day). If V2 (shelved) gets +20% recall on the pair tier vs main,
V2 unshelves immediately. If +<5%, V2 stays shelved with stronger
diagnostic justification. Either outcome is informative.

**Effort estimate.** Collection: 0 (synthetic). Curation: 1 day
(define adversarial configs). Integration: ~6 h (extend gen-synth
with multi-signal configs, new manifest schemas).

**Headline risk.** Overfitting to adversarial — graduation on
synth-adversarial doesn't transfer to real audio. Mitigate by
**always** requiring co-improvement on hard-200 or hard-1000 for
graduation; adversarial is a **diagnostic** tier, never primary.

## C4 — Continuous multi-hour single-band scored per-minute-window

**Description.** A single 24-hour continuous recording of one band
(say 20m or 14074 from K5ARH) sliced into 5760 consecutive 15 s
slots, scored not as 5760 independent WAVs but as a **time series**.
Metrics: per-15-min-window decode count, per-hour callsign diversity,
QSO completion rate (CQ → R → RR73 chains tracked across windows).
Visualization: heat-map of decode density vs time vs frequency.

Collection: K5ARH already captures continuously on Phase-5 deployment.
A one-week capture window yields ~7 days × 24 h × 240 slots/h ≈
40K WAVs. Sub-sample to one 24h day for the corpus.

**What it would reveal.** Whether decoder changes hurt or help
**temporal patterns** the per-WAV averages hide. Example: a change
that adds +1% mean recall but disrupts QSO chains (e.g., recovers
extra CQs but misses report-replies that depend on prior context)
would look like a win in hard-200 and a loss in this tier.

**Defensible prior.** wild_card: false. Time-series evaluation is
standard in radio-propagation research (ITU-R recommendations,
Chen et al. 2024 on solar-cycle-aware FT8 modeling). PSKReporter's
own UI defaults to time-series presentation.

**Assumption challenged.** "Slot-level recall is the right unit." It
isn't for autonomous operation — what matters is QSO completion
rate over an operating session.

**Kill-switch sketch.** Need ~24 h continuous capture (already
happening; just preserve it). Scoring infrastructure is non-trivial
(QSO chain extraction). Pilot with 1 h of data and 50-slot window
scoring; if patterns emerge that aren't visible in hard-200, scale
up.

**Effort estimate.** Collection: 0 (operator already capturing).
Curation: 1 day (slot inventory + manifest generation). Integration:
~3 days (new tier with windowed scoring, QSO chain extractor,
heat-map renderer). High but front-loaded.

**Headline risk.** Heavy infrastructure investment for unclear
hypothesis-unblock value. Justification requires evidence that
hidden temporal failure modes exist — pilot first.

## C5 — Operator-curated rare-DXCC tier (matters-to-K5ARH)

**Description.** Operator manually flags 100-200 slots from production
captures where a needed-DXCC or rare-grid station is present
(verified visually in WSJT-X or by callsign lookup). Each flagged
WAV is labeled with the rare callsign + true decode text. Recall is
measured **only on the rare-station decode**, not the surrounding
busy-NA ones.

Collection: 1-2 weeks of operator attention during normal operation.
K5ARH watches PSKReporter for rare signals appearing in his RX
audio, flags those slots.

**What it would reveal.** Whether decoder changes that improve
"average recall" actually move the needle on **mission-critical
decodes**. A change that adds +5% recall on common stations and
loses 1 rare DXCC decode is a deployment regression by user value
even though it's a hard-200 win.

**Defensible prior.** Operator-mission-value weighting is core to
radar/sonar evaluation (precision-on-target). FT8 DXing is similar
— one VK0 or P5 decode is worth 1000 W1AW decodes.

**Assumption challenged.** "All decodes are equally valuable." For
K5ARH's stated goals (cqdx priority scoring exists for a reason),
they are emphatically not.

**Kill-switch sketch.** Bootstrap: operator flags 20 rare-DXCC slots
over one weekend. If pancetta misses 5+ of them while WSJT-X
catches them, this tier has clear hypothesis-unblock value
(specifically: hb-064 DIA-OSD, hb-073 Doppler, structural-sync
ideas). If 0 misses, the tier is redundant with hard-200.

**Effort estimate.** Collection: 2 weeks operator attention (low
intensity). Curation: ~2 h to convert flags into manifest. Integration:
~4 h (rare-station-targeted recall metric in eval.rs).

**Headline risk.** Small sample (20-200 slots) is statistically
noisy. Mitigate by requiring large effect sizes (≥20% recall
delta) for any conclusion drawn from this tier.

## C6 — Pre-Costas-failure tier (zero-decode slots)

**Description.** Slots where pancetta produces ZERO decodes today.
Pull 200 such slots from a long capture (~few hours of typical 20m
operation will yield 50-100 zero-decode slots — slots where
propagation collapsed, deep QSB, or wideband interference). Truth =
jt9 + jtdx union (any decode counts). Recall measures "can we crack
a slot pancetta currently considers empty?"

Collection: scan production WAV archive for `pancetta_decode_count == 0`.

**What it would reveal.** The other side of the recall ceiling.
Hard-200 measures "given pancetta finds N, can we find N+δ?". This
tier measures "given pancetta finds 0, can we find anything?".
Failure modes here are upstream of LDPC/OSD — Costas detection,
spectrogram noise floor, sync threshold. Different hypothesis
class (hb-029-ish, hb-053-already-grad).

**Defensible prior.** Information-retrieval evaluation routinely
includes "hard negatives" — examples where the classifier returned
nothing. This is the FT8 equivalent.

**Assumption challenged.** "Recall improvements come from refining
existing decodes." Some recall comes from **finding signals at all**.

**Kill-switch sketch.** Sample 50 zero-decode slots; run jt9. If jt9
finds >5/50, the tier has signal — pancetta is leaving decodes on
the table from upstream (Costas/sync). If jt9 finds 0/50, those
slots are genuinely empty and the tier is redundant.

**Effort estimate.** Collection: ~2 h (scan + filter). Curation: ~2 h
(jt9 baseline + manifest). Integration: ~1 h (same as wild-100
shape).

**Headline risk.** Zero-decode slots may be genuinely silent — high
risk of building a tier that's noise. Pilot strongly required.

## C7 — High-jt9-novel-density tier (jt9-beats-pancetta inverse)

**Description.** Slots from production capture where `jt9_decode_count
- pancetta_decode_count >= 5`. These are slots where jt9 finds many
decodes pancetta misses — the inverse of pancetta-novel. Target
size: 200 slots. Truth = jt9 decodes. Recall = pancetta's coverage
of jt9's bigger find list.

Collection: re-score hard-1000 or full operator archive by panc-vs-
jt9 delta; pick top 200 by jt9-only count.

**What it would reveal.** The exact regime where pancetta is weakest
relative to a reference decoder. Hard-200 is biased toward slots
where pancetta does well (it was curated by pancetta scores). This
tier is biased the other way — directly measures the gap.

**Defensible prior.** wild_card: false. Stratified sampling against
the failure axis is standard in benchmark design (BIG-bench, GLUE
adversarial subsets, ImageNet-A).

**Assumption challenged.** "A pancetta-curated corpus measures
decoder performance fairly." It overcounts strengths. Adversarial-
to-baseline sampling exposes real gaps.

**Kill-switch sketch.** Build with existing data — survey already
computed jt9-only counts on 20 sample WAVs (174 jt9-only across
20 slots). Extend the survey to 1000 slots and pick the top 200.
Bootstrap is essentially free.

**Effort estimate.** Collection: 0 (existing archive). Curation:
~3 h (extend the survey scoring; pick top-200). Integration: ~2 h
(new manifest, eval dispatch).

**Headline risk.** Could be dominated by Costas-fail slots (C6
overlap). Mitigate by requiring `pancetta_decode_count > 0` so the
tier measures "ranking + LLR" rather than "sync".

## C8 — Multi-band simultaneous capture (cross-band consistency)

**Description.** Simultaneous K5ARH captures on 3-4 bands (20m + 40m
+ 15m + 17m) using parallel SDR receivers. 100 slot-aligned WAVs
per band, all from the same wall-clock instant. Each WAV truth via
jt9. Recall measured per-band but composite includes a "cross-band
consistency" sub-metric: when a callsign is decoded on band X, does
it also appear in the slot on bands where it's audible?

Collection: requires multi-receiver hardware. K5ARH has FTdx10 (one
RX); would need a KiwiSDR or SDR-Console with multi-RX, or
multiple Kiwi captures.

**What it would reveal.** Whether decoder behavior is band-stable.
Bands have different noise floor structure (40m broadcast intrusion,
15m absorption notch), and a decoder change that helps 20m may
hurt 40m. Hard-200 only measures one band.

**Defensible prior.** WSJT-X tests across bands implicitly via
operator reports; PSKReporter aggregates cross-band reception.
Academic radio papers often include multi-band evaluation.

**Assumption challenged.** "Decoder behavior is band-invariant; 20m
performance generalizes." It may not — band-specific noise
structures will exercise different decoder paths.

**Kill-switch sketch.** Pilot: one hour of dual-band 20m + 40m
KiwiSDR capture (cheap; KiwiSDR is the same data source as
hb-073). Compute decode count distribution per band. If
distributions are within 10% of each other on the same window,
band-invariance holds and the tier is low-value.

**Effort estimate.** Collection: ~1 week (operator sets up multi-RX
infrastructure or schedules Kiwi captures). Curation: ~6 h
(per-band manifests). Integration: ~4 h (cross-band consistency
metric in scorecard). Higher than most.

**Headline risk.** Equipment dependency. If multi-RX setup fails or
operator doesn't have bandwidth, this stalls indefinitely. Don't
schedule until C5/C7 are landed.

## C9 — Multi-station propagation diversity corpus

**Description.** Same QSO (same time, same TX station) captured at N
≥ 3 geographically diverse receivers (e.g., K5ARH in TX + a Florida
Kiwi + a California Kiwi + a NJ Kiwi). 100 slots × 3-4 receivers
= 300-400 WAVs. Each receiver's WAV is independent decoder input;
ground truth is the **union** of decodes across receivers (any one
RX that decoded the message → it's true).

Collection: PSKReporter coordinates multi-RX captures via the
KiwiSDR network. Operator scripts kiwirecorder.py against 3-4
public Kiwis simultaneously.

**What it would reveal.** Per-RX recall stability and noise-floor
sensitivity. A decoder change that helps low-SNR slots at K5ARH
might hurt high-SNR slots at the Florida Kiwi (different SNR
distributions). Multi-receiver-union truth is also a stricter
recall measurement — gives upper-bound on what's recoverable.

**Defensible prior.** wild_card: false. WSPRnet uses multi-receiver
diversity for propagation studies. RBN (Reverse Beacon Network) is
multi-receiver by design. Academic FT8 papers (Joe Taylor's own
documentation) cite multi-RX as truth ceiling.

**Assumption challenged.** "Single-receiver evaluation is sufficient."
Single-RX makes per-receiver noise structure invisible.

**Kill-switch sketch.** Pilot: one 15-min capture across 3 Kiwis +
K5ARH (4 receivers, 60 slots × 4 = 240 WAVs). If union-of-decodes
exceeds any single receiver by >30%, multi-RX truth is meaningfully
stricter and the tier is worth scaling.

**Effort estimate.** Collection: ~3 days (kiwirecorder cron + Kiwi
host selection). Curation: ~1 day (alignment + manifest). Integration:
~6 h (union-truth eval logic).

**Headline risk.** Receiver clock skew can misalign slots; KiwiSDR
public availability is unpredictable (servers go offline). Build
with at least 4 receivers so any one dropping doesn't kill a slot.

## C10 — Greyline-window targeted tier

**Description.** Capture WAVs only during the operator's local
sunrise (~12:30 UTC for TX in June) and sunset (~01:30 UTC in
June) windows — ±30 min each. ~120 slots per session × 7 days =
~840 WAVs. Greyline propagation has a distinctive Doppler signature
(rapid path-length changes as terminator sweeps).

Collection: scheduled K5ARH capture; trivial cron alongside
existing recording.

**What it would reveal.** Whether decoder behavior survives the
real-Doppler regime that hb-015 / hb-073 are designed for, BEFORE
hb-073's KiwiSDR-auroral capture lands. K5ARH greyline is
geographically accessible without operator travel.

**Defensible prior.** Greyline DXing is a documented FT8 phenomenon
(see KH6 DX traffic at NA sunrise); academic propagation papers
cite greyline as primary terminator-physics test case.

**Assumption challenged.** "synth-doppler captures real Doppler."
The corpus survey (2026-05-30) noted greyline as not yet observed
in K5ARH data — that may have been a one-session artifact.
Time-restricted capture forces the regime.

**Kill-switch sketch.** Capture 4 days × 2 windows × 30 min = ~480
slots. Compare jt9 decode count per greyline-slot vs midday-slot.
If greyline-slot decode count distribution differs by >20% (more
DX, lower median SNR), the regime is distinct and the tier is
worth building. If indistinguishable, drop.

**Effort estimate.** Collection: ~1 week passive capture during
greyline windows. Curation: ~3 h (filter + manifest). Integration:
~2 h (new tier, same shape as wild-100).

**Headline risk.** Greyline windows are short (30-60 min); seasonal
sunrise/sunset shifts mean the window moves; one week of data
might not yield a statistically clean separation. Acceptable: this
is observational, not adversarial.

## C11 — Contest-mode pileup corpus

**Description.** WAVs captured during a major FT8 contest weekend
(ARRL RTTY Roundup FT8 partition, ARRL DX, WAE) when the band is
saturated with serial-number exchanges and high TX rate. Different
message-content distribution (mostly contest exchanges, fewer
freeform CQs), much higher slot density, fast QSO rate.

Collection: scheduled capture during one contest weekend; ~48 h ×
240 slots/h = ~11K slots. Sub-sample to 200-500 for the tier.

**What it would reveal.** Decoder behavior at saturation. Hard-200
median is 24 decodes/slot; contest mode could push to 40+. Tests
NMS robustness, OSD load, joint-decoding scalability. Also: contest
messages have a distinct grammar (`<call> <call> <serial>`) — a
language-model-aware decoder hint (out of scope today) would show
up here first.

**Defensible prior.** WSJT-X release notes regularly cite contest
mode as a stress test. Contest robots (N1MM, etc.) demand high
decode rate; WSJT-X authors tune accordingly.

**Assumption challenged.** "Operator-typical density is the right
density." Contest density is the OS-equivalent of a stress test —
the regime where edge cases dominate.

**Kill-switch sketch.** Capture during the next available major
contest (ARRL DX FT8 plus is in September; CQ WW FT8 in early
December). Bootstrap window is multi-month — schedule as a
deferred capture. Until then, mock with C3 (high-collision
adversarial synth).

**Effort estimate.** Collection: 1 contest weekend (passive).
Curation: ~6 h (selection + manifest). Integration: ~3 h. Total
front-loaded by 1-3 month contest schedule.

**Headline risk.** Contest weekends are intermittent; can't iterate
quickly against this tier. Use as a quarterly checkpoint, not a
per-iter measurement.

## C12 — Bench-calibrated signal-generator corpus

**Description.** Use a signal generator + audio loopback to inject
WAVs with **precisely known** parameters: SNR ∈ {-26, -24, -22, ...
+10} dB at 1 dB steps, frequency ∈ {500, 750, 1000, ..., 2500} Hz
at 250 Hz steps, time offset ∈ {-0.5, -0.25, 0, +0.25, +0.5} s.
Each grid point gets 5 WAVs. Total: ~5000 WAVs, fully controlled.

Collection: requires audio signal generator (free; can use Python
soundcard generator) + clean acquisition path (loopback through
soundcard or pure synth). Operator hardware not strictly required
— this is synth with high-fidelity instrument calibration.

**What it would reveal.** Decoder behavior on a complete operating-
point grid without propagation confounds. synth-clean today is one
SNR sweep at a fixed frequency; this is a 3-D grid. Identifies
specific (SNR, freq, offset) cells where decoder fails — drives
targeted hypotheses.

**Defensible prior.** wild_card: false. RF receiver test methodology
standard practice — calibrated CW tone sweeps for IP3, NF, dynamic
range. WSJT-X authors use similar synthetic sweeps in
`wsjtx/lib/ft8/wsjt_test_modules`.

**Assumption challenged.** "synth-clean is the calibrated tier."
It's a 1-D sweep masquerading as calibration. Real calibration is
3+ D.

**Kill-switch sketch.** Build the generator (~1 day with existing
gen-synth infrastructure — extend to multi-dim sweeps). Run a
small 50-WAV pilot at (-20, 1000, 0) and adjacent points; if
decoder behavior is smooth/predictable, the tier is academic but
low-value. If there are sharp cliffs, the tier catches them
exactly.

**Effort estimate.** Collection: 0 (synthetic). Curation: ~1 day
(grid generator + manifest). Integration: ~3 h (new gen-synth
config schema). Compute: ~30 min for 5000 WAVs.

**Headline risk.** Pure synth — no propagation, no fading, no
real-world. Always pair with real-audio tiers for graduation
gating.

## C13 — Lid-of-band weak-signal-only corpus

**Description.** WAVs filtered to contain ONLY decodes at SNR ≤ -20
dB (jt9-reported SNR). Pull from large operator archive: every
slot where jt9 reports any decode at -20 dB or worse becomes a
tier entry. Target: 200 slots. Truth = jt9 decodes at SNR ≤ -20 dB
only (strong decodes in the same slot are ignored for recall scoring).

Collection: re-process production archive with jt9, filter by
SNR threshold.

**What it would reveal.** Decoder behavior at the noise floor. This
is where every PSKReporter spot of an exotic DX entity lives. The
metric is "of the signals jt9 sees at -20 dB, how many do we
recover?". A change that adds +5% mean recall but loses -20 dB
sensitivity is a lid-of-band regression.

**Defensible prior.** FT8 sensitivity is the **defining capability**
of the protocol vs FT4/JT9. Joe Taylor's original FT8 paper centers
on the -21 dB threshold. WSJT-X release validation tracks -20 dB
sensitivity explicitly.

**Assumption challenged.** "Mean recall is the right summary."
Lid-of-band recall is what differentiates FT8 from less-sensitive
modes. A decoder optimized for median SNR misses the FT8 mission.

**Kill-switch sketch.** Already a sub-population of hard-1000. Pull
the SNR-filtered subset; if it's <50 WAVs after filtering, broaden
to hard-200 + wild-100 + full archive. Pilot can be built in 1 day.

**Effort estimate.** Collection: 0 (existing archive). Curation:
~4 h (SNR-filtered manifest + per-decode truth tagging). Integration:
~3 h (recall metric needs decode-level filtering — not slot-level).

**Headline risk.** jt9-reported SNR is itself noisy (±2-3 dB at the
floor); filter threshold is fuzzy. Use a soft band (-21 to -19 dB)
rather than a hard cutoff.

## C14 — Continuous-capture metadata tier (antenna/band/SWR labels)

**Description.** Every K5ARH production WAV gets labeled with rig
metadata captured concurrently: antenna direction (rotator), band,
SWR, audio gain, ambient temperature, time-of-day-bucket. ~all
production WAVs (~thousands). Each labeled WAV stays in its
existing tier; this is an **annotation layer** for cross-tier
slicing.

Recall is sliced by metadata dimension: "recall on slots with
antenna pointing 0° vs 270°", "recall on slots with SWR > 1.5", etc.
Surfaces decoder sensitivity to **operating-context covariates**.

Collection: depends on hamlib/CAT plumbing to write per-slot
metadata sidecars during capture. Operator's coordinator already
has hamlib data flowing.

**What it would reveal.** Confounds in the existing corpus. Today
hard-200 silently mixes high-and-low-SWR slots, different antenna
headings, different bands. A "decoder improvement" that's actually
a "more decodes happen to be on the operator's good antenna" would
be invisible without this slicing.

**Defensible prior.** wild_card: false. Stratified analysis is
standard in epidemiology, A/B testing, and clinical trials.
Cohort-based slicing of a single dataset is cheaper than building
many small datasets.

**Assumption challenged.** "The corpus is i.i.d." It isn't — captures
are correlated with operator state.

**Kill-switch sketch.** Pilot: add metadata-sidecar writing to one
week of K5ARH capture; verify metadata is captured for >95% of
slots. If metadata coverage is low (rig comms drops, etc.), the
annotation layer is unreliable and the tier collapses to existing
hard-200. If coverage is high, slice hard-200 by metadata and
compare per-stratum recall; if strata differ by >10%, confounds
are real and the layer is valuable.

**Effort estimate.** Collection: ~3 days (sidecar writer + a week
of capture with metadata). Curation: ~6 h (sidecar schema + load
into eval). Integration: ~1 day (stratified recall in scorecard).

**Headline risk.** Hamlib data feed reliability — if CAT drops, the
sidecar is incomplete. Mitigate by treating missing-metadata slots
as a separate stratum, not by dropping them.
