## What shipped

`ContentFeatures` struct extended with 4 new `Option<_>` fields:
- `bp_iterations_used: Option<u8>`
- `osd_depth_used: Option<u8>`
- `nharderrs: Option<u8>`
- `min_llr_magnitude: Option<f32>`

New `content_score_v2_from_features` function with weights:
```text
v2 = v1
   + 0.20 * min_llr_magnitude
   - 0.01 * bp_iterations_used
   - 0.10 * osd_depth_used
   - 0.05 * nharderrs
```

Missing fields contribute zero; v2 reduces to v1 in that case. Existing autonomous-TX call site continues to use v1 (passes `None` for the new fields, byte-identical to pre-Batch-64 behavior).

3 new unit tests pin the contract:
- `v2_with_all_none_features_equals_v1`
- `v2_high_confidence_features_score_higher_than_low`
- `v2_partial_features_contribute_their_signal`

## Why v2 doesn't help

Three convergent reasons:

1. **The FP set on hard-200 isn't a convergence-quality set.** FDR's ConfidenceFeatures (and v2 by extension) are designed to discriminate "BP/OSD struggled" decodes from "BP/OSD breezed through" decodes. On hard-200, the FPs are CRC-passing-but-semantically-wrong messages (callsigns that don't exist, syntactically valid but bogus payloads). For these, BP/OSD telemetry is **identical** to TPs because the LDPC convergence is the same either way.

2. **v1's `confidence` field already encodes BP/OSD convergence.** The decoder sets `confidence = 1.0 - error_corrections / max_errors` (roughly). This is a downstream summary of the same signal `bp_iterations_used` would provide. v2 picks up at most a small marginal lift over the already-captured signal.

3. **The trust-set features dominate.** v1's `+2 if both_in_trust` and `+1 if any_in_trust` together account for most of the discrimination power on this corpus (probe seeded trust with 2736 callsigns). Once that's the dominant feature, BP/OSD telemetry is below the floor.

The v2 finding mirrors the FDR Session 4b finding (Batch 61): hard-200 doesn't exercise the regime where the mechanism helps. Same root cause; same outcome.

## Strategic implications

1. **ConfidenceFeatures infrastructure has yet to find a corpus where it pays off.** Both consumers (FDR module in Session 4b, hb-103 v2 here) are inert on hard-200. The infrastructure is correct and lightweight (~minimal hot-loop cost), so it stays shipped, but no consumer has produced a measurable win.

2. **The repeating pattern is real and important.** Every "advanced mechanism" investigation on hard-200 since Batch 53 has been measurement-negative. The corpus's signal distribution is too uniform — most decodes are clean Standard QSO traffic where BP converges fast and OSD doesn't run. The mechanisms that DO help (LLR whitening at Batch 53) operate at the LLR-input level, which IS exercised on every decode regardless of message-type or convergence-regime distribution.

3. **The next-tier productive work is not "more mechanism investigation."** Twelve consecutive batches of negative findings indicate the autonomous-loop search has converged on the existing corpus. Future positive findings require corpus changes (real-world capture for hb-244, FDR; synth-with-unusual-types for FDR), operational improvements (TUI work, coordinator polish, hamlib stability), or architectural pivots (audio-pipeline changes, signal processing rework at a different level than LLR refinement).

4. **The ConfidenceFeatures + hb-103 v2 ships at zero risk.** Both are byte-identical at the default code paths. Any future consumer that wants the telemetry can use it without paying any cost.
