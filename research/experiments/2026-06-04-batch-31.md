## Line A — pattern-based pre-rejects

| Pattern | hard-200 emit | hard-200 fp/tp | hard-200 truths | noise emit | FP rate | recall cost |
|---|---:|---:|---:|---:|---:|---:|
| self_call | 0 | 0/0 | 0 | 0 | n/a | 0% |

**Findings:**
- `degenerate_grid` (field letters outside A-R, or AA/ZZ extremes): 12 FPs eliminated at 0% recall cost — ship.
- `digit_run_callsign` (callsign with ≥3 consecutive digits): 1 FP eliminated, 0% recall cost — ship.
- `self_call`, `repeated_in_slot`, `same_callsign_twice`: 0 emissions on this corpus.
  - `repeated_in_slot` — already prevented by `unique_decoded` HashSet
  - `self_call` — pancetta doesn't emit our-call decodes on this corpus (no AP in eval)
  - `same_callsign_twice` — LDPC structure prevents this; would be an OSD/CRC-coincidence event

## Line B — message-content scoring

### Diagnostic R+S: per-feature AUC on full hard-200 + 50 noise

Collected per-decode features for 6969 samples (4937 TPs, 2032 FPs).
Per-feature Mann-Whitney AUC for TP/FP discrimination:

| Feature | AUC | Strength |
|---|---:|---|
| **in_trust_set (both callsigns)** | **0.837** | **strong** |
| **in_trust_set (any callsign)** | **0.755** | **strong (hb-062's criterion)** |
| **confidence** | **0.706** | **strong** |
| snr_db | 0.578 | weak |
| time_offset (inverted) | 0.576 | weak (FPs cluster at dt~8s vs TPs at dt~2s) |
| has_report | 0.565 | weak |
| frequency_offset | 0.530 | weak |
| text_len, has_grid | ~0.50 | uncorrelated |
| error_corrections | 0.500 | dead (zero variance in production eval) |
| ap_level | 0.500 | dead (AP not in eval default) |

**Surprises**:
- `confidence` (the decoder's own self-reported confidence) is a strong
  discriminator that wasn't previously used in the FP filter pipeline.
- `time_offset` shows FPs cluster around dt=8s (uniform-noise expected
  around dt=6s; pancetta's search biases late) vs TPs around dt=2s.
- `error_corrections` and `ap_level` are zero-variance — these fields
  are unused in eval, so they can't discriminate.

### Diagnostic T: fused score, AUC + operating curve

Hand-weighted fusion formula:
```
score = 2 * (both_callsigns_in_trust)
      + 1 * (any_callsign_in_trust)
      + 1 * confidence
      - 0.1 * time_offset
      + 0.05 * snr_db
```

**Combined AUC: 0.886** (strong discriminator).

Threshold sweep on full hard-200 + 50 noise (4937 TPs / 2029 FPs):

| Threshold | TP recall | FP reduction |
|-----------|----------:|-------------:|
| -0.07 | 100.0% | 17.2% |
| +0.35 | 100.0% | 34.3% |
| **+2.98** | **98.0%** | **72.9%** |
| +3.16 | 95.7% | 75.4% |
| +3.63 | 64.6% | 85.6% |

At 100% recall, the fused score is WORSE than hb-062 alone (17% vs
55-67% FP reduction). The fused score is only valuable above the
no-recall-loss point. This shapes the ship strategy.

### hb-103 v1 ship: `pancetta-qso/src/content_score.rs`

New module exposing:

```rust
pub struct MessageContentScore;
impl MessageContentScore {
    pub const SHIP_PRECISE: f64 = 2.977;     // 98% recall, 73% FP reduction
    pub const SHIP_AGGRESSIVE: f64 = 3.160;  // 96% recall, 75% FP reduction
    pub const SHIP_CONSERVATIVE: f64 = 0.352; // 100% recall, 34% FP reduction
}

pub fn content_score_from_features(
    feat: ContentFeatures<'_>,
    filter: &CallsignContinuityFilter,
) -> f64 { ... }

pub struct ContentFeatures<'a> {
    pub text: &'a str,
    pub confidence: f32,
    pub snr_db: f32,
    pub time_offset: f64,
}
```

**Critically: `CallsignContinuityFilter::accept()` is NOT changed.**
hb-103 v1 is a sibling scorer — consumers pick their threshold:

- **Autonomous TX decisions**: use `SHIP_PRECISE` (2% recall cost, 73%
  FP reduction). Prevents spurious TX initiation.
- **TUI display ordering**: rank by score for operator-facing priority.
- **PSKReporter / logging**: no threshold (recall preserved).

6 unit tests:
1. `both_in_trust_scores_higher_than_any` — score arithmetic
2. `untrusted_scores_below_trusted` — sign of trust contribution
3. `low_confidence_lowers_score` — confidence weight
4. `late_dt_lowers_score` — time-offset penalty
5. `ship_thresholds_are_ordered` — invariant
6. `unknown_callsign_scores_low_enough_to_reject_at_precise` — practical
   discrimination check

All 6 tests green; lib.rs exports added.

## Phase 3 — combined-stack impact (Diagnostic U)

On full hard-200 + 100 noise windows (4937 TPs / 2091 FPs):

| Stack | TP recall | FP reduction |
|---|---:|---:|
| no filter (baseline) | 100.0% | 0% |
| **hb-062 alone (currently in prod)** | **100.0%** | **55.3%** |
| hb-062 + patterns (redundant — already in `accept()`) | 100.0% | 55.3% |
| **hb-062 + hb-103 SHIP_PRECISE** | **98.0%** | **74.2%** |

The "patterns" arm is redundant because `accept()` already calls
`has_high_risk_fp_pattern()` as of Batch 30. The currently-shipped
production filter is at the 55.3% FP-reduction point.

The 74.2% point with hb-103 PRECISE is **+19 pp FP reduction at 2%
recall cost** — the Pareto choice for high-precision consumers
(autonomous TX, contesting). Not in default `accept()`; consumer-opted.

## Substance-check notes

- **Going deep paid off**. Batch 30 Q (just /R) gave 22% FP reduction.
  Batches 30+31 stacked: 55.3% at 100% recall; up to 74.2% at 98%
  recall. The "milk this line" instruction was right — there was real
  surface area.
- **AUC 0.886 is a strong empirical justification** for shipping the
  scorer. Not a soft "looks interesting" — quantified discriminator.
- **The 100%-recall arm of the curve shows hb-062 alone is already at
  the precision ceiling** that pure callsign-membership can reach.
  Beyond that point requires either (a) accepting recall cost, or
  (b) a structurally different feature (the message-content scoring
  path).
- **Default `accept()` semantics unchanged.** No production regression
  risk; the score is purely additive. Consumers that want higher
  precision opt in by importing and applying the score.
- **error_corrections (AUC 0.500)** is a dead field across the entire
  eval — every decode reports 0. Worth a future cleanup: either populate
  it or remove from the struct.
- **ap_level (AUC 0.500)** would discriminate if AP were used in eval,
  but the default `Ft8Config` has empty `ApContext`. Future iters wiring
  AP into eval might find this becomes a useful signal.
