## Method

### Prior-set aggregation

`CallsignPriorSet::from_session1_pool` reproduces Session 1's diagnostic
inputs exactly: operator=K5ARH, recent_this_wav extracted via the same
`callsigns_in` tokenizer, bundled-common=BUNDLED_COMMON_ACTIVE (75
entries, identical list), cqdx_spotted=empty (cqdx is not wired into
the eval harness; reserved for Session 3 plumbing).

`iter_unique(64)` returns the union in priority order (operator >
recent > cqdx > bundled), uppercase, deduped, capped at 64 entries.
This is the "≤64 per WAV" prior set the Session 3 mechanism would
iterate at each residual position.

### Micro-test design

The Session 2 spec envisioned LLR extraction at the truth's known
(freq_bin, time_step) coordinates directly from the post-multipass
residual, manual AP1/AP2 injection, and a manual LDPC+CRC+plausibility
funnel. That requires direct access to private pancetta-ft8 internals
(`Spectrogram`, `par_extract_symbols_from_spectrogram`,
`par_compute_soft_llrs_db`, the internal `LdpcDecoder`), which is out
of scope for Session 2 under the "no production decoder changes"
constraint.

The operational substitute runs the **production AP path end-to-end**:

1. Decode each top-20 hard-200 WAV with `decode_window` (AP off) →
   `ap_off_decoded` set of strings. Build `recent_this_wav` from the
   extracted callsigns (mirrors Session 1's aggregation and pancetta-
   qso's rolling-window logic).
2. Identify `missed_truths` = jt9 baseline messages not present in
   `ap_off_decoded` (loose match: either-contains-other).
3. Filter missed truths to those with at least one extractable callsign
   in `iter_unique(64)`. Pick the first 10 diverse-across-WAVs
   (per-WAV cap = 1 with fallback to 2; the cap prevents single-WAV
   monoculture — without it 9 of 10 picks came from WAV 0).
4. For each picked truth: build
   `ApContext { my_call: None, recent_calls: [RecentCallAp(truth_call)],
   active_qso: None }`. This activates the hb-043 my_call-less recent-
   only path
   (`par_try_ldpc_with_recent_only` in
   `pancetta-ft8/src/decoder.rs:~3715`) which injects the prior at
   BOTH bits 0-27 (caller) and bits 28-55 (called), runs
   LDPC+CRC+plausibility, and returns the better of the two attempts.
   Run `decode_window_with_ap(samples, ctx)` → `ap_on_msgs`.
5. Rescue = truth-in-AP-on AND NOT-in-AP-off. Record per-truth.

The test is a **lower bound** on Session 3's mechanism because it lets
Costas sync_search choose positions rather than enumerating residual
positions at known truth coordinates. Session 3 would extend the AP
machinery to positions sync_search missed; if AP+LDPC at sync-found
positions doesn't rescue prior-covered truths, the bypass-Costas
extension would inherit the failure (it pins the same 28 callsign bits
in the same LDPC code; the only difference is where the LLRs come
from, and noise-at-position is noise-at-position).

### Run

```text
cargo run --release -p pancetta-research --example hb087_session2_ap_decode_microtest
```

Two passes: (1) AP-off decode + prior-aggregation across 20 WAVs
(~3 min), (2) AP-on decode of 10 WAVs with singleton recent_calls
(~3 min). Total wall-clock ~6 min on this machine.

### Observations on the AP-on/AP-off equality

The AP-on and AP-off decode-count totals are identical for every WAV.
This is consistent with the my_call-less recent-only path acting as a
no-op on these specific residuals — `par_try_ldpc_with_recent_only`
returns Some only when CRC + plausibility both pass after injecting
the prior; on every picked truth, the injection produced no
CRC-passing convergence on any sync_search-surfaced candidate. AP did
not introduce FPs either (decoder did not surface garbage codewords
matching the injected prior callsign), which is the structural
guarantee — if there's no signal, BP cannot converge to a CRC-passing
codeword with the prior pinned at ±15.0 LLR.

A subset of these picks (4, 7, 8) appear as duplicate (callsign,
truth) tuples across WAVs (TN8GD KQ4KAO twice from neighbouring slots,
N4ML K8DID twice, N6NXV twice). The duplication is incidental — both
slots happen to carry the same QSO exchange — but it does mean the
"10 diverse WAVs" sample is effectively ~7 distinct truth messages.
Even at 7 distinct truths, 0 rescues is well below the ≥3 PROCEED
gate.

## Decision

**SHELVE.** 0 of 10 rescued (kill-switch ≥3 required).

The micro-test directly tests the structural assumption underlying
hb-087:

> "If we KNOW what callsign should be at this position (from priors),
> AP-constrained LDPC on the residual will rescue the truth."

It does not, on this corpus, for these 10 truths. The production AP
path injects the prior at exactly the bit positions hb-087 would
inject at, runs the same LDPC + CRC + plausibility funnel, and across
10 diverse picks rescues 0 truths. The bypass-Costas extension that
Session 3 would build does not change the LDPC code's behaviour on the
LLR vector — it only changes which positions the LLRs come from. If
the AP machinery can't rescue prior-covered truths at the positions
Costas DOES surface, it won't rescue them at the positions Costas
misses either, because the residual energy at sub-Costas positions is
WORSE (by definition Costas filtered them out for low sync_score).

## What this null result tells us about the broader bypass-Costas family

This result is structurally informative beyond hb-087:

1. **AP injection alone doesn't rescue sub-threshold residuals.** The
   ±15.0 LLR pin on 28 callsign bits is a strong constraint, but it
   doesn't manufacture signal where there isn't any. BP needs the
   remaining 146 non-pinned LLRs to point coherently in one direction
   for the codeword to close to CRC. If those LLRs are dominated by
   noise (which they are at sub-Costas-threshold residual positions),
   the prior can't pull BP across the gap.
2. **The sibling hypothesis hb-088 (OSD-without-Costas, SHELVED
   pre-impl 2026-05-31) reached the same structural conclusion via a
   different route**: OSD's flip enumeration also depends on the
   non-pinned LLRs having usable sign information, which they don't.
   Both bypass-Costas hypotheses fail at the same wall.
3. **The "missed truths whose callsign is in our priors" universe is
   not just below the Costas threshold — it's below the AP-rescue
   threshold too.** Session 1's 23.6% coverage is upper bound on
   *opportunity*, but the realised yield is essentially zero with
   either bypass-Costas constraint (callsign-prior OR OSD-flip).
4. **What WOULD work for these truths.** The structural failure mode
   says the residual at these positions is dominated by noise. A
   mechanism that improves the residual itself (better subtraction,
   coherent multipass refinement, multi-cycle accumulation across
   adjacent slots) is more likely to surface them. hb-079's coherent
   iterative-subtract was the biggest win in project history (+72 hard-
   1000 rec); future work in the same family (lookback-window
   coherent averaging across decoded-with-no-decode neighbours,
   per-position residual SNR estimation pre-decode) is the leverage
   the bypass-Costas family doesn't have.
5. **The V3-doctrine refinement worked as designed.** Geometric
   proximity (Session 1, 56.8% / 23.6%) is not decodability. The
   per-truth decodability gate (this session) catches what the
   coverage diagnostic missed. Session 3 implementation cost saved:
   the full mechanism (residual-position enumeration + per-prior LDPC
   sweep + FP funnel) would have been 1-2 days of work with negative
   return.

## Caveats on the SHELVE decision

The micro-test exercises the production AP path, not the idealised
"AP at known truth coords on the post-multipass residual" the spec
envisioned. Two-line argument for why the null result still
generalises:

- **Position is not the limiting factor for AP-rescue.** The
  production AP path tries the prior at every sync_search candidate,
  including positions within ±a-few-bins of every subtracted decode
  (because sync_search's local maxima cluster around real signals).
  If the truth's position is within sync_search's reach, the
  production path would have surfaced it. None did across 10 picks.
- **For the truths sync_search missed (Session 3's true target),
  the LLR-energy at those positions is lower** (Costas didn't find a
  pattern there — by construction the residual energy isn't
  forming Costas-like sync). AP injection on LOWER-energy residual
  LLRs cannot do better than AP injection on the SYNC-SURFACED-but-
  LDPC-failed residual LLRs we just tested. So the bypass-Costas
  extension can't improve on the null result.

A small residual possibility: a different parameter regime (heavier
multipass, more aggressive LLR variance target, different
plausibility gating) could change the AP-rescue probability. But the
project's production tuning is already saturating those knobs for
other hypotheses. Without a clear lever, SHELVE is the right call.

If hb-079-style coherent-subtract refinements later improve the
residual quality at sub-Costas positions, hb-087 could be revisited.
The aggregator + bundled list + micro-test infrastructure stays in
place and is reusable. The structural shelf-reason — "AP injection
doesn't rescue noise-dominated residuals" — would need to be falsified
by a corpus refresh or a downstream mechanism win that improves the
residuals these truths live in.

## What we learned about pancetta's residual LLR alignment with truth codewords

Indirect inference from the null result, not directly measured:

- At the **sync_search-surfaced candidates** the production decoder
  evaluates, LDPC with AP-pinned callsign bits cannot close to CRC for
  the prior-covered missed truths. Either the surface candidates are
  not at the truth's position (geometric miss) OR they ARE at the
  truth's position but the non-AP LLR sign agreement with the truth
  codeword is too low for BP to converge with the 28 AP bits pinned.
  Distinguishing these would require the direct LLR-extraction
  diagnostic the spec envisioned (Session 2's idealised version) —
  which would need pancetta-ft8 to expose either residual access or a
  diagnostic-only AP-at-known-position helper.
- The hb-088 OSD-without-Costas scoping (sibling, SHELVED 2026-05-31)
  has the LLR-alignment infrastructure already built (in-research
  spectrogram + Costas + LLR pipeline). A follow-up diagnostic could
  reuse that to measure truth-codeword sign agreement on these
  specific 10 truths; if alignment is <60%, "noise-dominated residual"
  is structurally confirmed and no AP-style mechanism will work.
  Logged as a future investigation but not blocking this SHELVE.
