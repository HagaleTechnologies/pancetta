## The mystery

Two main.json scorecards, both labeled "main" baseline on hard-1000
(identical corpus, 1000 WAVs, 28104 truth_decodes_total):

| baseline | rec | novel | composite | elapsed |
|---|---:|---:|---:|---:|
| OLD `research/scorecards/history/main.2026-05-30-pre-refresh.json` | 14987 | 3038 | 0.5694 | 2572s |
| NEW `research/scorecards/main.json` | 14993 | 6320 | 0.5791 | 1338s |

Recall stable (+6 rec; well within sub-thread-ordering noise). Decoder
defaults snapshot to byte-identical config (`sync_time_interp_delta_scale=0.3`
both, all hb-079/hb-080/hb-086 V1 flags on both). Eval tier list overlaps
exactly on hard-1000 (NEW also added `wild-100`, irrelevant to hard-1000).
Yet `novel_decodes` more than doubled — and the run was **half the wall-clock**.

The corpus-refresh ingestion journal
(`research/experiments/2026-05-31-corpus-refresh-ingestion.md` lines 91-113)
flagged this anomaly but punted root-cause to a follow-up. This is that
follow-up.

## Investigation

### Step 1 — Inspect both scorecards' decoder config snapshots

Both report:

```
"sync_time_interp_delta_scale": 0.3
"sync_time_interpolation": true
"joint_pair_retry": true
"coherent_multipass_iterations": 3
"layered_bp": true
...
```

Identical. The OLD scorecard's `head_sha` is `03516b8b` (hb-086 V1 grad
commit, 2026-05-28) with `dirty: true` — its worktree had the hb-068
delta_scale=0.3 changes applied as uncommitted edits. NEW has
`head_sha=0f8299e` (corpus survey commit, post-hb-068-graduated) with
`dirty: true`. So both runs effectively ran hb-068-style decoder code.
`residual_energy_stop_db: null` (hb-016, hypothesis 2) is a no-op at None
on both sides (the `Option` short-circuits the new probe inside
`coherent_subtract_and_repass` — zero behavioral change when off, verified
by reading the diff).

### Step 2 — Audit baseline-cache mtimes

Hypothesis 1 / 4 from the brief: did the corpus refresh regenerate any
hard-1000 baselines, shifting their decode sets? Script:

```python
import os, datetime, json
with open('research/corpus/curated/ft8/hard_1000.manifest.json') as f:
    shas = {e['wav_sha256'] for e in json.load(f)['entries']}
recent = sum(
    1 for s in shas
    if os.path.exists(f'research/baselines/ft8/{s}.json')
    and datetime.datetime.fromtimestamp(
        os.stat(f'research/baselines/ft8/{s}.json').st_mtime
    ) > datetime.datetime(2026, 5, 30, 12)
)
```

Result: **0 of 1000 hard-1000 baselines were modified post-refresh**.
The 197 baselines stamped `May 31 04:47` are 100 today-WAV
additions for the refreshed hard-200 plus 96 wild-100 generations + 1
overlap. None overlap hard-1000. `baseline_parallel` skips existing
baselines (`if out.exists() { return Ok((false, 0)); }` —
`examples/baseline_parallel.rs:106`), so no overwrites occurred. The
`hard_1000.manifest.json` itself has mtime `May 20 18:36` — untouched.

This kills hypotheses 1, 3, 4 in the brief: dedup state didn't shift, manifest
ordering didn't change, baselines weren't regenerated.

### Step 3 — Read the novel-counting code

`pancetta-research/src/bin/eval.rs:639-748` (`run_curated_tier`):

```rust
// Match: a baseline decode is "recovered" if our output contains it.
for truth_msg in &baseline_decodes {
    if our_decodes.iter().any(|d| d.message.trim() == truth_msg.trim()) {
        recovered_here += 1;
    }
}
truth_recovered += recovered_here;

// "Novel" decodes: ones in our output that aren't in baseline.
for ours in &our_decodes {
    if !baseline_decodes.iter().any(|t| t.trim() == ours.message.trim()) {
        novel_decodes += 1;
    }
}
```

`our_decodes` is whatever `decoder.decode_wav()` returned. Internally,
`Ft8Decoder::decode_wav` (`pancetta-research/src/decoder.rs:445-457`)
dedups by message text via a `HashSet`. So a duplicate decode within
a WAV does NOT inflate the novel count. The variable that DOES change
`our_decodes` between runs is the FP filter:

`pancetta-research/src/bin/eval.rs:484-491`:

```rust
fn apply_fp_filter(filter: Option<&pancetta_research::FpFilter>,
                   decodes: &mut Vec<Decode>) {
    if let Some(f) = filter {
        decodes.retain(|d| f.accept(&d.message, true));
    }
}
```

`apply_fp_filter` is called at `run_curated_tier:685` *before* novel
counting. **When `--fp-filter-baselines` is omitted, `filter = None` and
no decodes are dropped.** When the flag IS passed, `FpFilter::accept`
(`fp_filter.rs:224-258`) drops decodes whose callsigns are absent from
the baselines-union reference set. These dropped decodes are by
construction NOT in the truth-decode list either (or they'd be in the
baseline → in the filter set → not dropped), so dropping them is
*invisible to recall* but *removes that exact count from novels*.

But NEW `main.json` actually reports hard-200 `novel=1970`, not 1024.
**Recall (4942) matches**; novel (1024 vs 1970) does not. The V3 agent

## Root cause (ranked by confidence)

1. **(HIGH CONFIDENCE)** The two scorecards used different
   `--fp-filter-baselines` flag state. OLD = on, NEW = off. The FP
   filter drops decodes whose callsigns aren't in the baselines-union
   reference set; these dropped decodes are by definition not truth
   matches (recall invariant) but they DO count as novels when not
   dropped (the only delta).
2. **(LOW)** Slot-thread ordering / rayon nondeterminism (explains the
   ±6 recall noise but cannot explain a 3282 novel jump).
3. **(REFUTED)** Baseline-cache regeneration. 0/1000 hard-1000
   baselines were touched.
4. **(REFUTED)** Decoder default behavior change. `residual_energy_stop_db:
   None` is a no-op; all other defaults snapshot identically.
5. **(REFUTED)** Manifest reordering. hard_1000.manifest.json untouched
   since 2026-05-20.

## Is the +3282 novel "real"?

**It is a measurement artifact, not a decoder improvement.** Specifically:

- The NEW value (6320) reflects what the decoder actually emits when
  the harness applies *no false-positive filter*.
- The OLD value (3038) reflects what the decoder emits when the harness
  applies the strict-membership FP filter against the baselines-union.
- The decoder behavior is unchanged. The 6320 includes ~3300 marginal
  /low-confidence /likely-FP decodes that the OLD measurement quietly
  hid.

**Neither number is "wrong"**, but they answer different questions:

- novel_with_fp ≈ 3040 = "novels we'd ship if production used the FP
  filter today" (high-precision novels we can trust to be real
  unrecorded contacts).
- novel_no_fp ≈ 6320 = "raw decoder output that doesn't match a jt9
  baseline" (high-recall + larger FP tail).

The session log's "+703 hard-1000 rec, +240 hard-200 rec" cumulative
delta from 5 days of graduations IS still valid — those were all
measured against a fixed pre-hb-068 baseline using a consistent
flag profile (every sweep used FP filter ON per the hb-068 journal).
Recall numbers are robust to the filter flag; novel numbers are not.

## What to do about it

### Eval bug? No — design choice that's poorly documented.

The eval binary correctly applies (or doesn't apply) the FP filter
according to CLI flags. The "bug" is **methodology**: the canonical
`main.json` recipe in RUNBOOK.md omits the FP filter, but every
graduation sweep in `research/scorecards/sweep/*.json` and every iter
journal so far has been written with `--fp-filter-baselines` on. Two
incompatible measurement regimes were both labeled "main.json".

### Methodology fix (recommended)

Pick one of:

**(A)** Make the canonical `main.json` always use `--fp-filter-baselines
research/baselines/ft8`. Update RUNBOOK.md, the corpus-refresh tooling,
and document the rationale (production-relevant precision floor). This
keeps the "+703 hard-1000 rec" lineage continuous and gives sweep
agents a baseline they can A/B against without flag bookkeeping.

**(B)** Make the canonical `main.json` always use NO FP filter (raw
decoder output) and require every comparison to its own apples-to-apples
counterpart. This makes `main.json` a property of the decoder alone
(orthogonal to the filter), at the cost of a one-time discontinuity in
the historical-comparison chain.

**Recommended: (A).** Reasons:
- The 5-day session lineage of graduation deltas is FP-filter-on. Don't
  break that chain.
- Production pancetta already runs an FP-filter analog (the
  `CallsignContinuityFilter`); main.json should measure the
  production-relevant regime, not the raw decoder dump.
- All iter graduation gates (hb-068's "+5 rec / -7 novel") were
  evaluated FP-filter-on. Keeping main.json consistent means future
  graduation gates work without re-baselining.

### Concretely, propose to land (separate iter)

1. Add a `--fp-filter-baselines-default` config (or a guard flag) so
   that `main.json`-output runs are required-or-defaulted to FP filter
   on; output an explicit "fp_filter_active" field into the scorecard's
   `config` section.
2. Update `docs/RUNBOOK.md:355-358` canonical recipe to add
   `--fp-filter-baselines research/baselines/ft8`.
3. Re-run NEW main.json with FP filter on to produce a
   apples-to-apples baseline against the OLD lineage. Predicted
   composite shifts modestly downward (the snr_50pct_synth_clean and
   fixtures_pass_rate components are unaffected by the filter; the
   real_decode_rate_hard_200 component is unaffected too; the composite
   formula doesn't read novels at all, so re-running with FP-on will
   give the SAME composite within rayon noise). The hard-1000 novel
   number will fall from 6320 → ~3040.
4. Add an FP-filter-active assertion to the scorecard schema and the
   compare tool so future cross-baseline diffs can't silently mix
   regimes.

### Future-proofing on next corpus refresh

The corpus-refresh ingestion script
(`pancetta-research/examples/merge_corpus_refresh.rs`) doesn't invoke
eval directly — the human/agent runs it manually per RUNBOOK. After
fix (A) above, the recipe is self-documenting (the canonical command
includes the filter). Add to the corpus-refresh checklist:

> When refreshing the corpus, re-run main.json using the canonical
> recipe from RUNBOOK.md verbatim. Before overwriting the prior
> main.json, confirm `fp_filter_active` matches between the new run
> and the archived prior. Cross-check hard-1000 novels: refresh should
> not change them by more than ±100.

## Side-finding: scorecard schema is silent on FP filter state

`pancetta-research/src/scorecard.rs:ConfigInfo` records the decoder
config and the seed but **does not record CLI flags used for the run**.
Adding `fp_filter_active: bool` (and optionally `fp_filter_source:
String`) to `ConfigInfo` would make this entire class of "did the
methodology shift?" question diagnostic-by-default for future cycles.
Minor schema bump, but worthwhile. Logged as a follow-up here rather
than landing in this journal (which is read-only by mandate).
