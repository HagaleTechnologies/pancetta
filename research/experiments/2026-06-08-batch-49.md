## Process: 4 agents in parallel × shared checkout, then 2 agents × worktrees

### Wave 1 (shared checkout, with stash/HEAD-rebuild discipline)

Dispatched 4 implementer agents + 1 measurement agent in parallel into the same working directory. Each agent had clean-room firewall (specs only, no peer source).

| B | hb-226 Gaussian-ramp subtract | **NO COMMIT** — lost concurrent-edit race on decoder.rs |
| C | hb-230 relaxed sync + cycle smoothing | Committed via stash/HEAD-rebuild (incidentally discarded A's working-tree wiring) |
| D | LDPC feedback refinement | Committed via Python-splice/reset/reapply |
| E | hb-242 + max_sync_candidates sweep | Probe running in background |

Outcomes 3-of-4: C and D used self-isolation tactics (reset to HEAD, apply own diff, commit). B tried the same and gave up after the race continued.

### Wave 2 (worktree isolation, follow-up)

Re-dispatched A's wiring + B in `isolation: "worktree"` (each in its own `.claude/worktrees/agent-…` directory on its own branch). Both finished cleanly with **zero contention**.

But the worktrees forked from `04948df` (main), not from `iter/2026-06-08-batch-49` HEAD where Wave 1's C+D commits had landed. So cherry-pick onto iter had to navigate ~7 conflict regions (mostly in the shared `mod tests` block).

Resolution tactic: keep-both for `Ft8Config` struct + Defaults block (~2 conflicts), accept-ours for shared `mod tests` (~5 conflicts), then append B-rerun's tests as a fresh `#[cfg(test)] mod hb226_gaussian_ramp_tests` at end of file.

### Lessons captured

1. **Same-checkout parallel = bad** for files multiple agents touch. The stash/HEAD-rebuild discipline that C and D used worked but is brittle; B couldn't make it work and lost its work.
2. **Worktree parallel with correct base = good** if the fork base is the *current iter HEAD*, not main. Default `isolation: "worktree"` semantics fork from somewhere convenient for the tool, not necessarily for your batch.
3. **Shared `mod tests` block** is a primary conflict generator. Each agent appending tests at end-of-mod-tests collides with every other agent's append. Easy fix: instruct each agent to put its tests in its **own** `#[cfg(test)] mod foo_tests` block.
4. **Some serialization wins**: if 3+ agents add fields to the same struct (`Ft8Config` in our case), staging Wave 1 → land → Wave 2 (built on Wave 1's state) eliminates the struct-conflict entirely.

These go into a feedback memory.
