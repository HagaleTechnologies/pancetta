## hb-219 — /R → /P renderer (Item B + J)

### Phase B diagnostic

`batch35_probes.rs` Phase B: count /R, /P, /M emissions vs truths.

Pre-fix:
- Pancetta emits: **0 /P, 434 /R, 0 /M**
- jt9 truth has: **16 /P, 0 /R, 0 /M**

Pancetta hardcoded `/R` for ip=1 (encoder protocol-equivalent flag for
/R contest rovers AND /P portable). jt9's convention is /P. Pancetta's
choice missed all 16 /P truths because the rendered text didn't match.

### Fix

`pancetta-ft8/src/message.rs::parse_type1_standard`:

```rust
// Was:
Some(format!("{}/R", c))
// Now:
Some(format!("{}/P", c))
```

+ explanatory comment + 2 test updates:

1. `test_ip_flag_renders_as_slash_p_not_slash_r` (NEW) — verifies ip=1
   renders with `/P` suffix, not `/R`.
2. `tests/wsjtx_compat_tests.rs::test_suffix_round_trip` (UPDATED) —
   both /R-input and /P-input round-trip to /P-output (protocol is
   indistinguishable; pancetta now defaults to /P).

All workspace tests green.

### Measured impact

Post-fix audit:
- Pancetta emits: **434 /P, 0 /R, 0 /M**
- TPs: 5246 → **5253 (+7 TPs)** corpus-wide

The 7 /P TPs recovered are from jt9 truths whose text now matches
pancetta's rendering. The remaining 9 /P truths weren't decoded by
pancetta at all (sync failure or capture-effect).

### Compatibility with Batch 30 /R-filter

Batch 30 shipped `has_high_risk_fp_pattern` rejecting /R-suffix tokens.
That filter was effective because pancetta emitted /R for all ip=1
noise FPs (100% /R FP rate on hard-200).

Post-hb-219: pancetta emits /P instead. The /R-filter never fires.
The 434 noise FPs that previously got rejected by the /R-pattern now
flow into the callsign-continuity filter. Per Batch 31 Diagnostic F
(0/502 noise-FP callsigns in trust set), callsign-continuity catches
100% of them. **Net production impact: +7 TPs at ~0 additional FP
leak.** The /R-filter remains in place as defense-in-depth.

## Probes that didn't ship

### A — slot-edge silence-pad probe

Hypothesis: pancetta misses negative-dt truths because sync iterates
`t0 in 0..max_time_step` (can't search before WAV start). Pad each
WAV with 2s of silence at the start; re-decode.

Result on first 50 WAVs:
- Negative-dt truths examined: only 3 (rare slice)
- Default recovered: 2/3 (66.7%)
- Padded recovered: 2/3 (66.7%)
- **Padding lost 1269 TPs from non-negative-dt positions**

The 2s silence shift breaks the spectrogram alignment for all the
NORMAL signals at positive dt. Padding is the wrong fix.

Slot-edge sync expansion needs a decoder-internal change (e.g.,
allowing negative t0 in the Costas iteration). Not a one-line fix.

### C — multipass=2 probe

Re-run hard-200 with `max_decode_passes = 2`:

| Config | Decodes | TPs | Precision |
|---|---:|---:|---:|
| mp=1 (default) | 6828 | 5246 | 76.8% |
| mp=2 | 7156 | 5278 | 73.8% |

**+32 TPs gained from mp=2**, 0 TPs lost. Plus 296 additional FPs.

Batch 28's finding (mp=2 adds 0 TPs) is now obsolete after Batch 32
is_plausible changes and hb-217. New state: mp=2 adds +32 TPs.

The 296 additional FPs would flow through callsign-continuity. Per
Batch 31 F catching 100% of noise FPs at full trust, the effective FP
cost is near-zero. Tradeoff favors mp=2.

BUT: mp=2 also adds ~50% wall-clock cost (per Batch 28). For Fast tier
hardware (M4): acceptable. For Slow tier (Pi 4 class): probably too
expensive.

**Deferred decision**: candidate for hb-091 family revisit with the
new TP yield numbers. Mp=2 could ship as a tier-conditional default
(default ON for Fast tier, OFF for Slow tier, via the hb-216 hardware
classifier).

The hb-217 bug was the Maidenhead grid "RR73" (igrid4 = 32373) visually

collisions for the RRR or 73 tokens?

`unpackgrid` reservations:

- 32402: RRR (3 chars — no 4-char grid matches "RRR")

- 32404: 73 (2 chars — no 4-char grid matches "73")

2-char strings respectively — no 4-char Maidenhead grid can render to
those exact strings. No analogous bug for RRR or 73 tokens.
