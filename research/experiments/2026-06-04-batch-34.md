## Phase 1 — hb-217 RR73 fix

### Phase 1A: minimal reproducer

`batch34_rr73_reproducer.rs`: synthesize known-missed RR73 messages
in isolation, decode at various noise levels.

Result: **all synthetic RR73 messages decode at clean SNR and all
noise levels tested.** Code path works for isolated signals; the bug

### Phase 1B: real-WAV diagnostic

`batch34_rr73_internals.rs`: load a specific known-missed-RR73 WAV
(sha=17c4b25a, "WB3FME KB9AVX RR73" at 791 Hz / +14 dB SNR), decode,
and dump the full `Ft8Message` internals near each truth position.

**Result decisive**:

```
truth: WB3FME KB9AVX RR73 @ 791 Hz +1.30 s +14 dB
pancetta: WB3FME KB9AVX @ 791 Hz +2.00 s +8 dB
  message_type:    Standard
  standard_type:   Some(Reply)
  to_callsign:     Some("WB3FME")
  from_callsign:   Some("KB9AVX")
  grid_square:     None
  signal_report:   None
  contest_exchange:None
  payload[59..77]: 111111001110101001
  igrid4 (Std):    32373
  i3:              1
```

**Every missed RR73 has `igrid4 = 32373`** — not 32403 as expected for
the RR73 token. This is the Maidenhead-grid "RR73" string at field
17,17 (RR field, square 73).

### Phase 1C: ship fix

The existing parser (line ~1409, `parse_type1_standard`):

```rust
// Filter "RR73" from grids — it's a QSO-completion token, not a
// Maidenhead locator (igrid4=32373 collides with the RR73 token at
// igrid4=32403). Allow grids on both CQ and reply messages.
let filtered_grid = grid.as_ref().filter(|g| g.as_str() != "RR73").cloned();
```

The author **knew** about the collision but only filtered the RR73
out of the grid field — never promoted it to a token. So the message
became a "blank exchange" Reply with no grid, no report, no token →
displayed as just "WB3FME KB9AVX" with no trailing RR73.

**Fix**: when the grid filters out as RR73-shaped, synthesize the
token so the downstream branch (line ~1420, `else if let Some(tok) =
token`) sets `StandardMessageType::RR73`:

```rust
let (filtered_grid, rr73_from_grid) = match grid.as_ref() {
    Some(g) if g.as_str() == "RR73" => (None, true),
    _ => (grid.clone(), false),
};
let token = if rr73_from_grid && token.is_none() {
    Some("RR73".to_string())
} else {
    token
};
```

3 new unit tests:
- `test_rr73_via_igrid4_32373_collision` — the fix path
- `test_rr73_via_igrid4_32403_canonical_still_works` — original token path preserved
- `test_rr73_canonical_grid_passes_through_unchanged` — non-RR73 grids unaffected

All 165+ pancetta-ft8 lib tests + workspace tests green.

## Phase 3 — coverage bucketing

### 3A — recall by slot-position (dt)

| dt range | truths | recovered | recall |
|---|---:|---:|---:|
| 1.5..2.5 s | 5565 | 3483 | **62.6%** |
| 0.0..1.5 | 1875 | 1082 | 57.7% |
| <0.0 (early/late) | 1376 | 664 | 48.3% |
| 2.5..4.0 | 37 | 17 | 45.9% |

Pancetta's sync is centered on the slot middle; recall falls off at
slot edges. The early/late bucket at 48.3% is the largest recall gap.

### 3B — recall by frequency

| freq range | truths | recovered | recall |
|---|---:|---:|---:|
| ≥2500 Hz | 1124 | 802 | **71.4%** |
| <500 Hz | 865 | 554 | 64.0% |
| 2000..2500 | 1601 | 949 | 59.3% |
| 500..1000 | 1754 | 1003 | 57.2% |
| 1000..1500 | 1773 | 981 | 55.3% |
| 1500..2000 | 1736 | 957 | 55.1% |

**Counter-intuitive**: middle of band (1000-2000 Hz) has WORST recall.
The band extremes have BEST recall. Hypothesis: middle of band is the
busiest in real operations (most operators land there), so capture-
effect concentrates there.

### 3D — recall by callsign-1 length

| len | truths | recovered | recall |
|---|---:|---:|---:|
| 5 | 4117 | 2506 | **60.9%** (dominant) |
| 6 | 3277 | 1966 | 60.0% |
| 4 | 1238 | 714 | 57.7% |
| 3 | 6 | 1 | 16.7% (tiny sample) |
| ≥7 | 3 | 0 | 0.0% (compound callsigns, tiny sample) |

5-6 char standard callsigns dominate and decode well. The ≥7 char
hits are compound callsigns (/P, /M etc.) where pancetta misses
everything — a recall gap pancetta is aware of but not yet addressing.

### 3G — recall by neighbor density (capture quantification)

| neighbors within ±25 Hz / ±1 s | truths | recovered | recall |
|---:|---:|---:|---:|
| 0 | 4805 | 3652 | **76.0%** |
| 1 | 3259 | 1389 | 42.6% |
| 2 | 742 | 198 | 26.7% |
| 3 | 47 | 7 | 14.9% |

**Capture-effect dominates everything in the multi-neighbor case.**
Each additional neighbor cuts recall almost in half. With 0
neighbors, 76% recall is close to the decoder's clean-signal limit.
With 3+ neighbors, recall drops to 14.9%.

This is the corpus-scale capture-effect signature.
