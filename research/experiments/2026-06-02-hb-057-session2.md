## What this iter did

Implemented Session 2 of hb-057 per spec
`docs/superpowers/specs/2026-05-31-hb-057-median-dt-design.md`. Built a
per-callsign median+IQR DT prior store, threaded a `DtPriorLookup` trait
+ `Arc<dyn ...>` through the decoder, and gated a residual-sync candidate
filter on the prior windows of callsigns decoded in the prior pass.

Cited prior art: JTDX commit "use median filter in average DT calculation"
(Feb 2022) — the design rationale for median (outlier-robust) over mean
+ stddev, and the per-callsign rolling-window idea, both come directly
from JTDX. Window radius `max(0.2s, IQR * 3)` is pancetta-specific
(diagnostic-derived; the 0.2s floor matches the Session-1 ±0.2s gate).

## Methodology lesson — why V1 didn't fire

The Session-1 diagnostic measured a **38.6% recovery population**:
missed truths whose **sender callsign** has cross-WAV DT history with
stable/moderate variance. The narrowing mechanism the spec describes
(spec §"Wire into sync", line 152-159) hinges on the *target
candidate's callsign* being known so its prior can narrow the sync
window:

> "the multipass driver knows the callsign for each target position (it
>  was DECODED in the prior pass; that's how we have a callsign at
>  all). Pass the prior down with the target_positions vector."

This statement scopes to the V3 `localized_costas_sync_search` path
(SHELVED at default-off, never runs in production) AND to the AP path
(where the operator injects known callsigns). Production
`coherent_subtract_and_repass` step 2 runs a *full*
`costas_sync_search_with_threshold` over the residual — it surfaces
**unknown-callsign** candidate positions whose callsign is revealed
only by successful LDPC.

V1 as implemented filters residual candidates by the union of prior
windows from THIS-WAV's pass-1 decodes. This is a different hypothesis
from the spec's: it assumes missed truths' DTs cluster around the DTs
of already-decoded callsigns in the same WAV — true for some
band-aligned clusters, false for arbitrary independent operators. The
cross-WAV history IS maintained correctly, but the lookup key is
"callsign already decoded here" rather than "callsign we want to
decode here." With the wrong lookup key, the filter pulls priors that
don't predict where missed truths actually sit, and the residual
candidate set is narrowed in a way that mildly suppresses
random-FP-pattern decodes (-21 novels) but doesn't surface new
truth decodes (-1 recall).

The diagnostic-first kill-switch from Session 1 was a *gate-coverage*
upper bound — it measured what the prior CAN cover, not what the
filter would convert. Session 1 explicitly flagged this caveat (spec
§Diagnostic, line 42-47):

> "the recoverable = target-pop ceiling reflects the *gate's* coverage
>  rather than what LDPC will actually convert. The implementation
>  must measure real conversion in Session 2."

Session 2 measured real conversion at the chosen hook point: **zero
recall conversion** because the hook point doesn't expose the
target-candidate callsign at sync time.

## What to do next

The mechanism is NOT disproven — the population it targets exists
(38.6% of missed truths) and the per-callsign median+IQR is the right
statistic. What's disproven is the V1 hook point.

Three Session-3 candidates, in increasing implementation cost:

1. **AP-path hook** (cheap reuse of plumbing). When `ap_active` is
   true (operator sets `my_call` or `recent_calls`), the AP code path
   in `par_try_ap_decode` knows the candidate callsign upfront for AP2.
   Wire the DT prior to narrow the sync_search the AP path uses (per
   spec §"Wire into AP — additional detail"). Limitation: only fires
   when operator has AP context active, which is not the default eval
   condition (and the eval's main scorecards measure no-AP behavior).
   To eval this, change `--ap-rolling-window N` (hb-050 path) to
   build cross-WAV AP context AND DT history in lockstep. Honest
   bench: would require an AP-eval scorecard, not the production
   no-AP scorecard.

2. **Per-candidate callsign-keyed residual sync**. Re-architect
   `costas_sync_search_with_threshold` to accept an optional
   per-callsign prior set, and run the sync as one sweep per
   `(callsign, prior)` pair (each restricted to that callsign's
   prior window). Each pair surfaces candidates that would AP-decode
   as that callsign at that position. Significant decoder change;
   probably 2 sessions.

3. **Reject not narrow** (out-of-scope-for-V1 per spec §Out-of-scope,
   line 254-256). After a residual candidate decodes to callsign C
   with t0=T, if C has a prior with median far from T, reject as a
   probable FP. This is a *post-decode* filter, not a sync narrower.
   Could pair with hb-062 (FP filter) as an additional callsign-
   conditional check. Mechanism kill-switch needed first (what % of
   FPs have a callsign with a prior whose median is far?).

Priority recommendation for the bank: keep hb-057 at 0.40 (Session-1
PROCEED stands; mechanism not falsified), but split into hb-057a (AP
hook, cheap V2) and hb-057b (per-candidate sync, expensive V3). Per
the iter workflow, both are scoped before either implements.

## Methodology caveat captured during implementation

The spec's narrowing logic (V1 §"Wire into sync") targets either (a) the
V3 `localized_costas_sync_search` path (currently SHELVED at
default-off) or (b) the AP path where the candidate callsign is known
in advance. Production `coherent_subtract_and_repass` step 2 runs a
*full* `costas_sync_search_with_threshold` over the residual — it
surfaces NEW candidate positions whose callsigns are unknown until
LDPC succeeds.

V1 as implemented filters those residual candidates by the union of
prior windows from THIS-WAV pass-1 decodes. This is a weaker hypothesis
than the spec's "per-callsign prior narrows that callsign's sync":
it assumes that missed truths' DTs cluster around the DTs of
already-decoded callsigns in the same WAV (which holds for
band/network-aligned clusters but not for arbitrary independent
operators). The cross-WAV history *is* maintained so the multi-WAV
prior reflects what the Session-1 diagnostic measured, but the lookup
key is "callsign decoded in this WAV's pass 1" rather than "callsign
we're trying to decode here" (the latter requires AP-path context).

If V1 graduates, this is a clean win on the targeted-pair scenario
(hb-086 V1's same population). If it shelves, Session 3 should
re-implement with the AP-path hook OR per-candidate-callsign
narrowing (would need re-plumbing the residual sync sweep to be
callsign-keyed).
