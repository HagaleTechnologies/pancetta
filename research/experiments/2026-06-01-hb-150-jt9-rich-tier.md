## Motivation

The existing `curated-hard-200` tier is selected by *pancetta's own*
difficulty heuristic (curate-time `interest_score`, dominated by
pancetta_decode_count + noise floor + low SNR). That biases the tier
toward WAVs pancetta finds interesting, not toward WAVs that exhibit
pancetta's worst recall gap vs jt9.

mr-009 corpus ideation (entry C7 → hb-150) proposed inverting the
selection axis: curate by `jt9_novel_density = jt9_decode_count -
pancetta_decode_count_on_same_wav`. WAVs where this is large are the
ones jt9 finds meaningfully more decodes in than pancetta — i.e. the
denominator for any future sync / LLR / ranking change that targets the
pancetta-vs-jt9 gap.

This tier directly stresses pancetta's recall ceiling vs jt9 and
unblocks (or sharpens the eval signal for):

- The **hb-015 family** (Doppler-resilient sync search and related sync
  audits). hb-150's WAVs surface the slots where pancetta finds far
  fewer signals than jt9, which is exactly the regime hb-015 is supposed
  to attack.
- **Bias-detection work** (e.g. callsign-prior false-positive audits):
  having a high-volume jt9-truth tier makes per-WAV recall vs precision
  trade-offs visible at scale.

Zero new audio required — we already have 1317 baselined WAVs in
`research/baselines/ft8/`. The work is pure curation + eval-dispatch
plumbing.

## Mechanism

### Curation binary

New binary `pancetta-research/src/bin/curate_jt9_rich.rs` (registered as
`curate-jt9-rich` in `pancetta-research/Cargo.toml`).

Inputs:

- `--baselines-dir research/baselines/ft8` — directory of jt9 baseline
  cache files. Each `<sha>.json` provides the jt9 decode count for that
  WAV.
- `--manifest-snapshot <path>` (repeatable) — existing curated manifests
  (`hard_1000.manifest.json`, `wild_100.manifest.json`) whose
  `score_breakdown.pancetta_decode_count` is harvested to avoid
  re-decoding every WAV.
- For any baselined WAV not present in any provided snapshot, the
  binary runs the current `Ft8Decoder::with_default_config()` against
  it and counts decodes.

Scoring: `jt9_novel_density = jt9_decode_count - pancetta_decode_count`.
Sort descending; take top-N (default 200). Emit a `CuratedManifest` at
`research/corpus/curated/ft8/hard_jt9_rich_200.manifest.json` with the
same schema as the existing manifests (`hard_200.manifest.json`,
`wild_100.manifest.json`).

Caveat documented: the `pancetta_decode_count` for the 1100 WAVs taken
from existing snapshots is the curate-time count (2026-05-31), which
predates a few weeks of decoder graduations (e.g. hb-086 graduated
2026-05-29 — before manifest, ok; later iters may also matter). The
relative *ordering* of the gap signal is robust across small decoder
deltas, so the selection is stable. The eval pipeline runs the *current*
decoder against jt9 truth so the *baseline recall number* reported below
is live-accurate; only the manifest-membership decision uses the
snapshot.

### Eval-tier dispatch

New `"hard-jt9-rich-200"` arm in `pancetta-research/src/bin/eval.rs`,
modelled on the `"wild-doppler-50"` stub: missing manifest → log +
`TierResult { wavs_processed: 0, .. }`. Once the manifest exists, the
arm hands off to `run_curated_tier` exactly like the other curated
tiers.

Help text updated; tier list now includes `hard-jt9-rich-200`.

### Composite metric

hb-150 is INFRA. The composite metric (`pancetta_research::metrics`)
is unchanged: `hard-jt9-rich-200` is *not* a composite term, so adding
the tier cannot move the composite score by itself. Future decoder
changes evaluated against this tier will get a per-tier
`decode_rate` / `vs_wsjtx_pct` to read; the composite weight discussion
is left to a follow-up if the tier proves useful.

## Baseline eval (current default decoder on hard-jt9-rich-200)

Scorecard: `research/scorecards/sweep/hard-jt9-rich-200-baseline.json`.

| Metric                | hard-jt9-rich-200 | curated-hard-200 (reference) |
|-----------------------|------------------:|-----------------------------:|
| WAVs                  |               200 |                          200 |
| jt9 truth decodes     |             9385  |                         8853 |
| pancetta recovered    |             4151  |                         4942 |
| **decode_rate**       |        **44.23%** |                   **55.82%** |
| novel_decodes         |               640 |                         1024 |
| eval wall-clock (s)   |             177.6 |                            — |

**Punchline:** the new tier sits **~11.6 percentage points lower** in
recall than the existing curated-hard-200, even though both contain
200 WAVs. The curation worked: hard-jt9-rich-200 stresses the
pancetta-vs-jt9 recall axis directly, exactly as the design called
for. Future decoder changes targeting sync resilience (hb-015 family)
or LLR-side ranking will have a clean denominator here.

The 4151-recovered / 9385-truth split implies ~5234 missed truth
decodes on this tier — a ~5x larger headroom than what
`curated-hard-200` exposes (8853-4942 = 3911 missed).

## Disposition

SHIPPED. Pure infrastructure — no decoder change, no composite move,
no fixtures perturbation. The lift is **future-enabler**: any
sync-resilience / Doppler / LLR / ranking experiment can now ship a
`hard-jt9-rich-200` number alongside its `hard-200` number and read
both ends of the difficulty axis. hb-015 family + future bias-audit
work pick this up automatically.
