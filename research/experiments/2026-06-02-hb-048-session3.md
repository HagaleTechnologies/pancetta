## Headline

**Session 3 lands a7 production wiring (decoder pass + Ft8Config fields +
CLI flags) and runs a 6-point threshold sweep on hard-200 with bootstrap
CI. Every threshold combination (snr7 ∈ {5.0, 5.5, 6.0, 6.5, 7.0},
snr7b ∈ {1.5, 1.8, 2.2}) produced rec Δ ∈ {0, +1} with the 95% bootstrap
CI straddling 0 — NOT statistically distinguishable from same-config
noise. Novel Δ was significant-positive everywhere (+382 to +727 over
the baseline 1024, all CIs well above zero). The mechanism injects
false positives without recovering truths on this corpus. SHELVED.**

## Session 3 scope (from design spec §"Session 3")

1. Wire `a7_cross_correlation_pass` into `decode_window_with_ap` after
   the V1 joint-pair-retry pass.
2. Threshold sweep over snr7 ∈ {5.0, 5.5, 6.0, 6.5, 7.0} and
   snr7b ∈ {1.5, 1.8, 2.2}.
3. Per-WAV scorecards + bootstrap CI on hard-200.
4. Graduation decision per Phase A.

## What was built

### Production wiring (`pancetta-ft8/src/decoder.rs`)

- `Ft8Config::a7_enabled: bool` (default false).
- `Ft8Config::a7_snr7_threshold: f64` (default 6.0 — WSJT-X reference).
- `Ft8Config::a7_snr7b_threshold: f64` (default 1.8 — WSJT-X reference).
- `Ft8Config::a7_freq_window_hz: f64` (default 6.25 — one pancetta
  freq_bin; WSJT-X uses 2 Hz).
- New private method `Ft8Decoder::a7_cross_correlation_pass` invoked
  from `decode_window_with_ap` right after V1 joint-pair-retry. See
  `pancetta-ft8/src/decoder.rs:1095` (hook) and the
  `a7_cross_correlation_pass` impl above the V3 method (around
  line 2690).

**Mechanism**: for each successfully-decoded callsign C in this window,
generate ~32 templates via `a7::generate_templates`. For each
`sync_candidate` within `±a7_freq_window_hz` of C's audio frequency
(and not at an already-subtracted position), extract residual LLRs via
`par_extract_symbols_from_spectrogram` + `par_compute_soft_llrs_db`,
run `a7::best_template_score`, accept the winning template's text via
`Ft8Message::from_text` when `snr7 ≥ threshold` AND `snr7b ≥ threshold`.
`is_plausible()` is checked downstream; the production FP filter runs
after `decode_window`.

### CrossTimeState bridge — deferred

The design spec called for threading `Arc<RwLock<CrossTimeState>>` into
`Ft8Decoder` at construction. `CrossTimeState` lives in `pancetta-qso`;
`pancetta-ft8` has no dep on `pancetta-qso` (and adding one would
introduce a circular relationship via the coordinator). Since the eval
harness creates a fresh `Ft8Decoder` per WAV anyway (eval is one window
at a time, no cross-slot context), the within-WAV path uses callsigns
decoded in the CURRENT window as the expected-call source. Production
cross-slot enrichment (reading from `CrossTimeState`) is a Session 4
follow-up if within-WAV graduates — which, per the sweep below, it
does not.

### CLI flags (`pancetta-research/src/bin/eval.rs`)

- `--a7-enabled` / `--no-a7`
- `--a7-snr7-threshold <f64>`
- `--a7-snr7b-threshold <f64>`
- `--a7-freq-window-hz <f64>`

Each threshold/window flag implicitly turns the pass on. Builder methods
also added on `pancetta_research::decoder::Ft8Decoder`.

## Decision per Phase A

**SHELVED.**

Mapping against the design spec's SHELVE criteria:
- hard-200 rec < +5 across the full threshold sweep: ✓ (max rec Δ is +1
  at the most permissive setting).
- "no plausible parameter set survives": ✓ (rec CI straddles 0 at every
  point in the 5×3 grid we tested, including the most permissive corner).
- composite delta is essentially zero at every setting (the +0.0001 at
  snr7=5.0/snr7b=1.5 is below the CI noise floor of the recall input).
- Novel Δ is significant-positive everywhere — FP injection without
  upside.

Mapping against the design spec's GRADUATE criteria:
- composite ≥ +0.0005 minimum: ✗ (max observed is +0.0001).
- hard-200 recovered +5 minimum: ✗ (max is +1).

This satisfies the design spec's SHELVE-definitive condition: "hard-200
rec < +5 across the full threshold sweep AND no plausible parameter set
survives."

## Why this differs from Session 2's GRADUATE

Session 2 ran a **synthetic-injection micro-test**: an explicitly-known
truth template (K1ABC W1AW RR73) is encoded to clean LLRs at known
positions and the cross-correlation primitive recovers it at snr7=65
even at noise SNR -7.6 dB. That demonstrated the *primitive correctness*
of `cross_correlate` / `best_template_score`, which is necessary but not
sufficient for the eval-corpus test.

What the production eval reveals:

1. **The within-WAV path generates templates from callsigns that
   already decoded in the same window.** The very-strong-correlation
   case (the residual at C's own position) is filtered out by
   `subtracted_positions`, but nearby `sync_candidate`s within ±6.25 Hz
   still get evaluated. When those positions hold *another* callsign's
   transmission (the structural case a7 is supposed to rescue), the
   template bank rooted at C is the *wrong* bank — the message there
   is some other station's traffic, not C's follow-up. The
   cross-correlation produces noisy peaks against the wrong-bank, and
   the FP filter catches them as novels, not as truths.

2. **The cross-WAV / cross-slot path is what WSJT-X's a7 actually
   tests.** WSJT-X stores callsigns heard in slot N and tests templates
   against slot N+1 — a *different* spectrogram of *different* audio,
   where C is likely the responder, not a third party at the same
   freq. Pancetta's eval can't simulate that (each WAV is independent;
   `prev_slot_calls` is empty). The CrossTimeState plumbing landed
   2026-06-01 specifically to carry this state across slots in
   production — but the Session 3 spec explicitly anticipated this gap
   (§Open-Q-1, §Risk-2) and budgeted to ship within-WAV only and
   measure that.

3. **Within-WAV a7 has a structural ceiling much lower than the
   cross-slot mechanism.** Within one 12.64s window, after multipass
   subtract has already cleaned C's strong decode out, the remaining
   residual at nearby freqs contains *other* signals with their own
   callsign payloads — not C's. The template-correlation prior is
   wrong for those positions.

## What this rules out (and what it doesn't)

**RULED OUT (binding):**
- Within-WAV a7 with the current decoder's residual-after-V1 surface.
  No threshold combination from the design-spec sweep grid recovers
  truths at this corpus's noise level.

**NOT RULED OUT (deferred):**
- Cross-slot a7 in production (CrossTimeState-seeded recent_calls fed
  to the decoder from a previous slot). This is the WSJT-X-original
  mechanism. To test it we'd need either (a) a chronological synthetic
  corpus of consecutive-slot WAVs where slot N+1 contains a known
  follow-up to slot N's transmission, or (b) a production trace.
- a7 with a different residual surface. The current pass runs against
  the post-V1 residual; running against the original spectrogram
  (pre-multipass) would have access to more SNR but also to all the
  already-decoded signals. The trade-off is unclear and out of scope
  for this iter.
- a7 templates rooted at the operator's own callsign (production
  context). Eval has no `my_call`, so the template bank lacks the
  "I expect a reply to ME" priority weight that's the WSJT-X-original
  highest-value case.
