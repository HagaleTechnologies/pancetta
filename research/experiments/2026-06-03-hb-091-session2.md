# hb-091 — a8-style early-decode latency reduction (Session 2)

**Date**: 2026-06-03
**Branch**: iter/2026-06-03-hb-091-session2
**Status**: **SESSION-2-COMPLETE — PROCEED to Session 3 at HW=±5 bins (±31 Hz)**
**Effort**: ~5 hours (primitive + tests + HW=2 diagnostic 44min wall + HW=5 diagnostic 162min wall)

## Session 1 recap

Session 1 (2026-06-02) ran a truncation-feasibility kill-switch on hard-200
without any scoping. At t=14.0s (1.0s early, the a8 sweet spot): retention
= 97.73% of full recall, Δ = −112 (95% bootstrap CI [−141, −80]). Cleared
the 95% PROCEED gate by 2.7 percentage points.

The Session 1 measurement is an **upper bound** on a8's recall: it uses
the full Costas sweep at the truncated length. Production a8 will also
restrict the sync sweep to a narrow `freq_bin_range` around the in-QSO
partner's known frequency, which trades recall for compute (smaller search
space = faster decode = the actual latency win).

## Session 2 question

Does scoped-search recall at t=14.0s clear the 95% gate vs full-search
recall at t=15.0s?

If YES → the operational a8 path (scoped decode at t=13s alongside the
standard t=15s pass) is viable. Session 3 builds the DSP-side partial-buffer
emit + coordinator wiring + loopback QSO/hr A/B.

If NO → scoping costs too much recall. SHELVE the feature; keep the
primitive gated default-off so future hypothesis work can use it without
the production fast-path going through it.

## Method

### Decoder primitive

Added `decode_window_scoped(samples, freq_bin_range)` and
`decode_window_with_ap_scoped(samples, ap_ctx, Option<RangeInclusive<usize>>)`
to `pancetta-ft8::Ft8Decoder`. The scope is threaded through both
`costas_sync_search` (initial sweep) and `coherent_subtract_and_repass`
(residual multipass), so a scoped decode never silently re-searches outside
the requested range. Existing `decode_window_with_ap` becomes a thin
wrapper passing `None`.

Tests (`pancetta-ft8/src/decoder.rs`):
- `test_scoped_decode_within_range_recovers_message` — synth FT8 at 2000 Hz
  (freq_bin 320), scoped 318..=322 → CQ recovered.
- `test_scoped_decode_excludes_out_of_range_message` — same signal,
  scoped 78..=82 → zero matching decodes.

Both pass; the broader pancetta-ft8 suite (~350 tests across all profiles)
stays green.

### Scoped-recall diagnostic

`pancetta-research/examples/hb091_session2_scoped_recall.rs`. For each WAV
in `hard_200.manifest.json` (200 real off-air recordings; aggregate 8853
jt9 truths from `research/baselines/ft8/<sha>.json`):

1. **recall(15.0s, full)**: standard 15s decode, count truths recovered.
2. **recall(14.0s, full)**: truncate to 14.0s, full decode (Session 1 replication).
3. **recall(14.0s, scoped)**: for EACH jt9 truth, derive its freq_bin from
   `truth.freq_hz`, run an independent `decode_window_scoped(samples[..14.0s],
   (truth_bin − 2)..=(truth_bin + 2))`, and count THAT truth as recovered
   iff its message string appears in the output.

Scope half-width = ±2 freq_bins ≈ ±12.5 Hz, the journal's "±10 Hz around
partner's last known freq" rounded up to the bin grid.

Bootstrap 95% CI on per-WAV deltas vs the 15.0s baseline via
`pancetta_research::bootstrap_recall_delta` (N=1000, seed=091_2026_06_02).

### PROCEED gate

- retention(scoped @ 14.0s) ≥ 95% of recall(full @ 15.0s), AND
- 95% bootstrap CI on Δ(scoped @ 14s − full @ 15s) excludes > 5% loss
  (`CI_low > -0.05 * baseline_recovered`).

This is the SAME gate as Session 1, just applied to the operational
scoped+truncated arm.

### Notes on the measurement

- The diagnostic uses jt9 truth's `freq_hz` as the scope center. In
  production, the coordinator stores the partner's `frequency_offset`
  from the previous decode; that estimate has ±2-3 Hz of jitter vs jt9.
  So the diagnostic measures an **upper bound** on scoped recall — true
  production performance is at-best the same, more likely a small
  fraction lower depending on freq-estimate variance.
- An independent `Ft8Decoder` is constructed per truth. Residual-multipass
  state must not leak across scoped passes that target different
  partners; the per-truth decoder enforces that cleanly.

## Results — HW=±2 bins (±12.5 Hz, journal-default)

Full hard-200 run, 43.8 min wall, 8853 jt9 truths total.

```
        arm  recovered     truths     recall
  ----------------------------------------------
  full @ 15.0s       4942       8853    0.5582  (baseline)
  full @ 14.0s       4830       8853    0.5456  (Session 1 replication)
  scoped @ 14.0s     4648       8853    0.5250
```

Δ vs full @ 15.0s baseline:

| arm | Δ | retention | 95% CI(Δ) |
|---|---:|---:|---|
| full @ 14.0s | −112 | 97.73% | [−141, −80] |
| scoped @ 14.0s | **−294** | **94.05%** | **[−331, −261]** |

PROCEED gate verdict:
- retention(scoped @ 14.0s) = 94.05% — **FAIL** (gate ≥ 95%)
- CI_low = −331 vs floor of −247.1 (= −5% of baseline) — **FAIL** (gate CI_low > floor)

**HW=±2 verdict: SHELVE.**

Marginal cost decomposition:
- Truncation alone: −112 decodes (−2.27% of baseline)
- Truncation + scoping at ±2: −294 (−5.95%)
- **Marginal scoping cost at HW=2: −182 decodes (−3.68%)**

The marginal scoping cost (−182) is larger than the truncation cost
itself (−112). Striking: ±12.5 Hz around the **jt9-truth-anchored** scope
center costs more recall than chopping 1 second off the buffer. And the
diagnostic uses truth-perfect anchoring — production scoping (anchored
on the partner's previous decode) would be at least as bad and likely
worse.

## Width-sweep follow-up (HW=±5 bins, ±31 Hz)

Before committing the SHELVE, an open question: is the HW=2 cost a
**width-tuning** issue (jt9 vs pancetta freq convention; sub-bin jitter
in pancetta's Costas pick; ±12.5 Hz tighter than partner-freq tracking
variance in real ops) or a **fundamental scoping cost** (LDPC + multipass
need more candidate diversity than 5 freq positions provide)?

A HW=±5 (±31 Hz) sweep distinguishes:
- If retention(HW=5) ≥ 95% with CI clearing the floor → width-tuning;
  PROCEED to Session 3 with HW=5 as the production scope width.
- If retention(HW=5) still < 95% → fundamental cost; SHELVE on both
  widths.

Diagnostic now accepts `--half-width N` (added 2026-06-03).

## Results — HW=±5 bins (±31 Hz)

Full hard-200 run, 161.5 min wall (HW=5 increases per-truth scoped
search by ~2.2x relative to HW=2; the diagnostic does N independent
scoped decodes per WAV, so this scales linearly in `(2·half_width + 1)`).

```
        arm  recovered     truths     recall
  ----------------------------------------------
  full @ 15.0s       4942       8853    0.5582  (baseline)
  full @ 14.0s       4830       8853    0.5456  (Session 1 replication; bit-exact match)
  scoped @ 14.0s     4746       8853    0.5361
```

Δ vs full @ 15.0s baseline:

| arm | Δ | retention | 95% CI(Δ) |
|---|---:|---:|---|
| full @ 14.0s | −112 | 97.73% | [−141, −80] |
| scoped @ 14.0s | **−196** | **96.03%** | **[−226, −167]** |

PROCEED gate verdict:
- retention(scoped @ 14.0s) = 96.03% — **PASS** (gate ≥ 95%, clears by 1.03pp)
- CI_low = −226 vs floor of −247.1 — **PASS** (clears floor by 21 decodes)

**HW=±5 verdict: PROCEED.**

Marginal cost decomposition (HW=±5):
- Truncation alone: −112 (−2.27%)
- Truncation + HW=5 scoping: −196 (−3.97%)
- **Marginal HW=5 scoping cost: −84 decodes (−1.70%)**

vs the HW=2 marginal scoping cost (−182, −3.68%), widening from ±12.5 Hz
to ±31 Hz **cuts the marginal scoping cost by more than half** —
mechanistically consistent with the ±12.5 Hz scope being barely covering
the jt9-vs-pancetta freq-convention slack + Costas sub-bin jitter, NOT a
fundamental decoder cost.

The "width tuning" interpretation is correct. ±5 bins is the right
production width.

## Decision: PROCEED to Session 3 at HW=±5 bins (±31 Hz)

The HW=5 width-sweep clears both clauses of the PROCEED gate on
truth-anchored hard-200, so the operational a8 path is recall-viable.
Session 3 = production wiring.

### Caveats (carry into Session 3 design)

- **Truth-anchored vs partner-anchored**: this diagnostic uses jt9's
  `freq_hz` as the scope center, which is the **upper bound** on
  production performance. In production, the coordinator anchors on the
  partner's PREVIOUS DECODE's `frequency_offset`, which has ±2-3 Hz of
  estimation jitter on top of the jt9-vs-pancetta convention slack.
  Production retention will be a bit lower than 96.03% — likely
  95.0-95.5%. Still passes the gate but close; Session 3 should measure
  partner-anchored retention on hard-200 as part of the loopback A/B.

- **QSO/hr is the actual operational gate**, not recall. Session 3's
  PROCEED-to-production-default-on requires QSOs/hr +≥10% under at
  least one fade scenario in the loopback A/B. If recall passes but
  QSO/hr doesn't move, SHELVE the production wiring; keep the primitive
  gated default-off.

### Session 3 scope

1. **Coordinator partial-buffer emit**: DSP-side change to deliver the
   first 13s of the slot as a partial buffer to the coordinator at
   t=13s into the slot. Today the DSP only delivers the full 15s
   buffer at slot end. This is the load-bearing plumbing change.

2. **Coordinator scoped-decode dispatch**: when `activeQso` is set
   AND a t=13s partial buffer arrives, call `decode_window_scoped` with
   `freq_bin_range` = partner's last `frequency_offset` ± 5 bins
   (mapped via `(freq_hz / 6.25).round() as usize`). If a
   partner-relevant message decodes, advance the QSO state machine
   immediately. The standard t=15s full-window decode still runs after.

3. **Loopback A/B harness**: extend `pancetta/tests/loopback_qso.rs`
   with a slot-timing model so QSO/hr is measurable. Vary partner
   fade timing across at least three scenarios (no-fade, slow QSB,
   fast fade). Measure: mean turnaround per leg, QSOs/hour, retention
   regression vs full-window decode.

4. **Default-off gate**: until the loopback A/B clears +≥10% QSO/hr,
   the scoped path stays default-off behind a config flag. The
   coordinator wiring is in place but inert.

### Session 3 expected effort

~2-3 sessions:
- Session 3a: DSP partial-buffer plumbing + smoke that partial buffer
  decodes the in-progress partner.
- Session 3b: Coordinator dispatch + state-machine integration.
- Session 3c: Loopback A/B harness + QSO/hr measurement + ship/SHELVE
  decision.

## Lessons

1. **The journal's "±10 Hz" heuristic was 1-2 bins too tight.** ±12.5 Hz
   (HW=2) cost −182 decodes; ±31 Hz (HW=5) cost only −84. The recall
   cost was width-tuning, not fundamental scoping cost. Sub-bin jitter
   in the Costas pick + freq_osr=2 sub-bin resolution + jt9-vs-pancetta
   convention slack add up to needing ≥3 bins of slack on each side.

2. **Two-arm comparison answers width-tuning vs fundamental** cleanly.
   Without HW=5, the right interpretation of the HW=2 SHELVE would have
   been ambiguous. The hb-091 hypothesis would have been incorrectly
   shelved at this stage if Session 2 had only measured HW=2.

3. **Cost: 161 min for HW=5 vs 44 min for HW=2.** Per-truth scoped
   sweeps scale with `(2·half_width + 1)`, so HW=5 is ~2.2x more work
   per truth than HW=2. For width-sweep diagnostics, schedule the
   widest arm first and most-of-the-day budget accordingly.

4. **Recall is a feasibility gate; QSO/hr is the deployment gate.** This
   session settled recall feasibility. Session 3 measures the
   operational gain. Don't confuse "the recall arithmetic works" with
   "the feature ships."

5. **TDD discipline paid for itself.** The first test failed in a way
   that exposed the BASE_FREQUENCY = 1500 Hz modulator offset — a
   one-off but real frequency-convention bug in the test setup that
   could have masked the entire diagnostic if pushed without verifying
   the failure mode. The "watch it fail correctly" step caught it
   immediately.
