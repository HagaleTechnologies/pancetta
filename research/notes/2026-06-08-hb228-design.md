# hb-228 design: JTDX 3-method spectral sweep in pancetta

Batch 44 Tier 3: implementation path for shipping hb-228 (highest-
priority new bank entry from Batch 43 JTDX research).

## What JTDX does (from Batch 43 agent finding)

`lib/sync8.f90`:
```fortran
! ipass=1,4,7 — sqrt(re² + im²)   amplitude
s(i,j) = SQRT(real(cx(i))**2 + aimag(cx(i))**2)

! ipass=2,5,8 — power (re² + im²)
s(i,j) = real(cx(i))**2 + aimag(cx(i))**2

! ipass=3,6,9 — L1 norm |re| + |im|
s(i,j) = abs(real(cx(i))) + abs(aimag(cx(i)))
```

Per-method `syncmin`: 1.225 / 1.5 / 1.1. Three different metrics
produce three different Costas-sync candidate lists from the SAME FFT.
Different signals pop under different metrics (amplitude favors
fading recovery; power-squared boosts strong SNR; L1 robust to
impulse noise).

## What pancetta does today

`pancetta-ft8/src/decoder.rs::compute_spectrogram` (line 1671):
```rust
let cval = fft_buffer[src_bin];           // complex FFT bin
let mag2 = cval.norm_sqr();                // power = re² + im²
let db = 10.0 * (1e-12f64 + mag2).log10(); // dB-power
row.push(db);
```

Single magnitude map: **dB-power** (= `10 * log10(re² + im²)`).

`min_sync_score=3.0` calibrated for dB-power units.

## Implementation paths

### Path A — Full integration (PROD ship target)

Modify `Spectrogram` struct to hold N power tensors:

```rust
pub struct Spectrogram {
    pub power: Vec<Vec<Vec<f64>>>,           // current: dB-power
    pub power_sqrt: Option<Vec<Vec<Vec<f64>>>>,   // NEW: sqrt(re²+im²)
    pub power_l1: Option<Vec<Vec<Vec<f64>>>>,     // NEW: |re|+|im|
    // existing fields...
}
```

Modify `compute_spectrogram` to populate the alternates when a config
flag is on:

```rust
pub struct Ft8Config {
    /// hb-228: enable JTDX-style 3-method spectral sweep. Builds
    /// alternate magnitude maps (sqrt, L1) alongside the default
    /// dB-power and runs Costas sync on each, unioning candidates.
    pub enable_3method_sweep: bool,
    // ...
}
```

Modify `costas_sync_search_with_threshold` to accept the alternate
maps and produce candidates from each, with calibrated per-method
syncmin (need empirical tuning).

Modify the decode driver to union candidates from 3 sweeps.

**Effort estimate**: ~400-600 LOC + tests + per-method syncmin
calibration. **2-3 sessions.**

### Path B — Research-only probe (cheaper validation)

Approach: build a custom decoder loop in `pancetta-research/examples/`
that:
1. Reads WAV
2. Computes 3 spectrograms (one per magnitude) — duplicate code from
   `compute_spectrogram` with parameterized magnitude function
3. For each: run a custom Costas sync (also duplicated from pancetta-ft8)
4. Union candidates
5. Use pancetta-ft8's public LDPC decoder to verify each candidate

**Effort estimate**: ~300 LOC duplicated. **1 session if interfaces are
stable.**

Risk: the duplicate Costas sync diverges from pancetta-ft8's
production behavior over time. Mitigation: extract Costas sync into
a `pub fn` in pancetta-ft8 (small refactor) so research example calls
the SAME code with different magnitude inputs.

### Path C — Pre-compute magnitudes externally, feed via API extension

Approach: add a public API to `pancetta-ft8`:
```rust
impl Ft8Decoder {
    pub fn decode_with_external_spectrogram(
        &mut self,
        spectrogram: Spectrogram,
        ap: &ApContext,
    ) -> Ft8Result<Vec<DecodedMessage>>;
}
```

Then in research example, compute 3 spectrograms manually, call
`decode_with_external_spectrogram` for each, union results.

**Effort estimate**: ~200 LOC in research + ~50 LOC API surface in
pancetta-ft8 + spec docs. **1 session.**

This is the cleanest probe path: minimal pancetta-ft8 changes,
clear research-only behavior.

## syncmin calibration concern

JTDX's syncmin values (1.225 / 1.5 / 1.1) are calibrated for raw
(non-dB) magnitudes. Pancetta's 3.0 is dB-power. Direct lift won't
work. Per-method calibration is required:

1. Pick a method (e.g., sqrt)
2. Generate synth signal at known SNR
3. Measure Costas sync_score distribution at various sync thresholds
4. Pick threshold that matches pancetta's dB-power's recall/FP trade

This is empirical work. Could be done via small synth-corpus probe.

Estimated calibration effort: 1 session per method × 3 methods = 3
sessions if done separately. Or a single 3-method joint calibration:
1.5 sessions.

## Recommended next-batch action

**Path C**: add `decode_with_external_spectrogram` public API to
pancetta-ft8 (~50 LOC + tests), build research example using it. This
lets us measure hb-228's headroom without committing to integration.

If measurement is positive (Δ > 30 TPs on hard-200), then justify
Path A full integration in a Batch 45-46 plan.

If measurement is negative or marginal, shelve hb-228 like Batch 43
shelved hb-224.

## Conflict + interaction analysis

- vs `hb-079/080 mp=2` (shipped): ZERO conflict. Sweep is sync-stage;
  mp=2 is residual-stage.
- vs `hb-086 V1 joint_pair_retry` (shipped): ZERO conflict.
- vs `hb-044 Costas parabolic` (shipped): NO direct conflict.
  Parabolic refines AFTER sync; sweep generates more candidates BEFORE
  refinement. Stack naturally.
- vs `hb-221 multi-interval` (deferred): unclear interaction. Both
  surface more candidates pre-LDPC. Could compound or could overlap
  on the same recovered truths. Measure independently first.

## Risks

1. **syncmin calibration is hard** — different magnitudes have
   wildly different sync_score scales. Bad calibration = either no
   gain or huge FP explosion.
2. **3x FFT-side compute** (or 3x memory for cached complex FFT).
   Acceptable on Fast tier; expensive on Slow.
3. **Downstream FP filters might break** under the additional
   candidate volume. Need to measure FP rate explicitly.

## Decision (Batch 44)

DEFER hb-228 implementation to Batch 45. Reason: requires
pancetta-ft8 API surface change + syncmin calibration; not a clean
single-batch ship.

Batch 45 plan should include:
1. Add `decode_with_external_spectrogram` public API
2. Implement Path B/C research example with 3-method sweep
3. Calibrate syncmin per method
4. Measure on hard-200
5. Ship-or-shelve decision based on Δ TPs and FP cost
