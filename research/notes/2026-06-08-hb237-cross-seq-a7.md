# hb-237 Cross-sequence A7 — Session 1 design

Batch 45 Tier 3: design the cross-sequence A7 extension of pancetta's
existing a7 module (hb-048 Session 2 shipped).

## Pancetta's existing a7 (hb-048 Session 2, shipped)

`pancetta-ft8/src/a7.rs::A7Template`:

- After slot N decodes callsign C at (freq f, time t), generate small
  set of "follow-up" templates rooted at C — e.g., `<C OTHER>`,
  `<C OTHER> R+10`, `<OTHER C> RR73`.
- Each template → FT8 encoder → 174-bit LDPC codeword (expected bits).
- Cross-correlate expected codeword against CURRENT slot's residual
  LLR stream via `cross_correlate()` returning `snr7` + `snr7b`.
- Mechanism is **within-slot**: C decodes in slot N → templates apply
  to slot N's residual.

Production wiring: Session 3 (status unclear; may not be fully wired).

## wsjtr's cross-sequence A7 (per Batch 44 agent finding)

`bodiya/wsjtr docs/cross_sequence_decoding.md`:

- Maintain `prev[Even][N]` and `prev[Odd][N]` decode tables indexed by
  sequence (parity).
- At each new window: take previous **opposite-sequence** decodes
  (if current slot is even, use last odd slot's decodes as seeds).
- For each seed callsign C: generate up to **206 candidate messages**:
  - `CALL1 CALL2` variants
  - `+ RRR/RR73/73`
  - `Report -50..+49`
  - Hashed `<call>` variants for non-standard suffixes
- For each candidate message: downsample current window at each
  (DT, freq), do fine sync + 4 LLR metric variants, correlate.
- **Acceptance criteria**: `dmin ≤ 100 AND dmin2/dmin ≥ 1.3` (best
  candidate must beat second-best by 30%).

WSJT-X has shipped this since v2.6.0 (Jun 2022). Implementation in
mainline ft8b.f90.

## Key differences pancetta vs wsjtr

| Aspect | pancetta a7 | wsjtr cross-seq A7 |
|---|---|---|
| Source slot for seed callsigns | CURRENT slot's decodes | PREVIOUS opposite-parity slot's decodes |
| Template count per seed | small (~10-20?) | up to 206 |
| Target slot for matching | CURRENT slot's residual | CURRENT slot's residual |
| State across slots | none | parity-tagged callsign cache |
| Score function | snr7 + snr7b (best/2nd ratio) | dmin + dmin2/dmin ratio |
| Production wiring | partial (Session 3 unclear) | shipped in WSJT-X 2.6.0+ |

## Mechanism intuition

Cross-sequence A7 exploits the **QSO exchange structure** explicitly:
when a station completes a CQ → grid → report → RR73 sequence, each
message lives in the OPPOSITE parity slot from the previous one. So a
station heard in slot N (parity X) is overwhelmingly likely to be in
slot N+1 (parity ~X) sending a different exchange-stage message.

Knowing the callsign from slot N gives strong AP for slot N+1's
templates. This is much stronger AP than pancetta's within-slot
"if we decoded C, also try follow-ups in the same slot".

## Pancetta integration design (Session 1)

### Required state

Add to pancetta-ft8 (or pancetta-qso?):

```rust
pub struct CrossSequenceCallCache {
    /// Callsigns decoded in the most recent even-parity slot.
    even_calls: Vec<RecentCall>,
    /// Callsigns decoded in the most recent odd-parity slot.
    odd_calls: Vec<RecentCall>,
    /// Cap on age — older entries pruned.
    max_slot_age: usize,
}

pub struct RecentCall {
    callsign: String,
    freq_hz: f64,
    dt_s: f64,
    snr_db: f64,
    slot_parity: u8,
    slot_number: u64,
}
```

This belongs in **pancetta-qso** — it's QSO-context state, not pure
decoder state. The pancetta-ft8 decoder accepts the cache as an input.

### Template expansion

Extend pancetta-ft8/src/a7.rs to expose:

```rust
pub fn generate_cross_sequence_templates(
    seed_call: &str,
    qso_context: &CrossSequenceContext,
) -> Vec<A7Template>;
```

Generate up to 206 templates:
- For each seed C in opposite-parity cache:
  - `C OTHER` (with several "OTHER" candidates from QSO context)
  - `C OTHER GRID` (with common grid suffixes)
  - `C OTHER -SNR` for snr in {-50, -45, -40, ..., +45, +49}
  - `OTHER C RRR`, `OTHER C RR73`, `OTHER C 73`
  - hashed `<CALL>` variants for compound

Need careful enumeration — 206 is the upper bound; in practice many
will be skipped (e.g., "OTHER" is a placeholder requiring a known
QSO partner).

### Cross-correlation step

Reuse existing `cross_correlate()` in pancetta-ft8::a7.

### Score gate

Pancetta currently uses `snr7` + `snr7b`. wsjtr uses `dmin` + `dmin2/dmin`.

Adopt the wsjtr-style **best/second-best ratio gate**: only accept a
candidate if it beats second-best by 30% (`dmin2/dmin ≥ 1.3`). This
matches WSJT-X's robustness criterion.

### Coordinator-level wiring

`pancetta/src/coordinator/`:
- After each slot decode, push decoded callsigns into the
  CrossSequenceCallCache tagged by current slot parity.
- Before next slot decode, pass cache to decoder as AP context.
- Cache lifetime: ~2-3 slot ages = ~30-45s.

This is a NEW data flow direction in pancetta. Currently:
- decoder → coordinator → QSO state machine

After hb-237:
- decoder → coordinator → cross-seq cache
- cross-seq cache → decoder (NEW; feeds next slot's a7)

## Expected delta

Per agent: "~30% of pancetta's response-shaped misses recoverable".
If pancetta misses ~3500 truths on hard-200 of which maybe 25% are
response-shaped (report / report_r / RRR / RR73 / 73) = 875 misses.
30% recovery = ~260 TPs. Highly speculative.

Conservative estimate: ~50-150 TPs depending on how well QSO context
predicts the opposite-parity slot.

## Risk assessment

- **Risk: AP context misalignment**. If pancetta decodes a callsign C
  in slot N but C wasn't actually in slot N (FP), hb-237 will inject
  FAKE expected templates into slot N+1. Could amplify FPs.
- **Risk: template explosion**. 206 templates per seed × multiple
  seeds × multiple slots = many cross-correlation evaluations. May
  exceed CPU budget on Slow tier.
- **Risk: cache management complexity**. Slot timing, parity, age
  tracking adds state that pancetta currently doesn't have.
