# hb-238 OSD `dmin` audit — finding: NOT A BUG, structural difference

Batch 45 Tier 2: audit pancetta's OSD `dmin` initialization per
Bodiya's wsjtr `docs/osd-depth-enhancement.md` finding.

## Bodiya's documented finding (wsjtr context)

Bodiya found that wsjtx's `osd174_91` seeds `dmin` (best-distance
tracker) from order-0's computed distance regardless of CRC pass/fail.
Bodiya's first wsjtr port initialized `dmin = INFINITY` and accepted
ANY CRC-passing codeword, giving ~4.1% noise FP rate per OSD call.

The bug pattern: "INFINITY init + any CRC-pass wins" → noise FPs
that happen to satisfy CRC-14 get accepted.

WSJT-X's fix: seed `dmin` from order-0's distance even if CRC failed.
This forces later candidates to BEAT order-0's distance AND pass CRC.

## Pancetta's OSD design (pancetta-ft8/src/osd.rs:233-466)

Pancetta's OSD has **no `dmin` tracking AT ALL**. The control flow:

```rust
// OSD-0 (0 flips):
if let Some(result) = self.try_solution(&info_hard, &base_parity, &final_perm) {
    return Some(result);  // ← RETURNS IMMEDIATELY on first CRC pass
}

// OSD-1 (1 flip):
for flip in 0..LDPC_INFO_BITS {
    if let Some(result) = self.try_solution(&info, &parity, &final_perm) {
        return Some(result);  // ← RETURNS IMMEDIATELY
    }
}

// OSD-2, OSD-3: same pattern
```

`try_solution` (line 433): compute CRC-14 on candidate codeword. If
calculated == received → return Some(codeword). Else None.

**Pancetta's policy: "lexicographically first CRC-pass wins"**.
No distance comparison across candidates.

## Why pancetta works despite different design

- Order of evaluation: 0 flips → 1 flip → 2 flips → 3 flips
- Most signals are correctly received → OSD-0 succeeds first
- Hard cases require flips; first-flip-that-passes wins
- **Downstream FP filters** (hb-062 callsign-continuity, hb-103 content
  score, is_plausible) catch the bad cases where "first CRC-pass" was
  a wrong candidate

The downstream filters are tuned to the current "first wins" behavior.
Per Batch 31 F: hb-062 catches 100% of noise-FP callsigns at full trust.

## Is pancetta's design WORSE than WSJT-X's?

In theory yes — "first wins" can accept a CRC-pass that's farther from
the received LLRs than another CRC-pass that the algorithm would have
found later. The "best by distance" approach is more robust at the
OSD layer.

In practice — unclear. Pancetta's measured precision on hard-200 is
~75% which is similar to other decoders. The downstream filters
compensate for the simpler OSD layer.

## hb-238 outcome: CLOSED, not actionable

The Bodiya `dmin INFINITY` bug does NOT apply to pancetta. Pancetta's
OSD design is structurally different and arguably simpler. Switching
to "best by distance" is a research-level redesign, not a 30-min audit
fix.
