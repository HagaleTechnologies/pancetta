## Ranked candidates

### 1. Subcode Ensemble Decoding (SCED / aSCED) for FT8 LDPC(174,91) — hb-261, 0.52

**Technique**: Run K parallel BP decoders on K *coset-shifted* versions of the
received LLR vector. Each shift s_k ∈ GF(2)^n is a random coset representative
of a GF(2) linear subspace; flipping the LLR sign at position i where s_k[i]=1
moves the channel observation into a different subcode of the same code, breaking
BP's trapping sets in a different way for each k. Collect the union of all
CRC-passing decodings.

**Primary sources**:
- arXiv:2501.11993 (January 2026) "Subcode Ensemble Decoding of Linear Block
  Codes" — Monte-Carlo on three LDPC codes: SCED improves over standalone BP
  AND over automorphism ensemble decoding (AE). Does NOT require the NP-complete
  automorphism group search.
- arXiv:2604.06889 (April 2026) "Affine Subcode Ensemble Decoding of Linear
  Block Codes" — extension with affine subcodes; near-maximum-likelihood
  performance on BCH codes using ≈ 64 BP decoding paths.

**Measured evidence**: 0.2–0.3 dB over standalone BP on CCSDS / 802.11n / 5G
LDPC codes (automorphism ensemble, arXiv:2202.00287); aSCED achieves near-ML
on BCH(128,64) and BCH(64,45) with K=64 paths, outperforming existing ensemble
schemes.

**Applicability to pancetta**: Excellent.
- FT8's (174,91) LDPC automorphism group is unknown; SCED bypasses this
  requirement entirely (key advantage over AE decoding, arXiv:2503.01738).
- Implementation fits rayon: K independent BP calls with perturbed LLR input,
  union of CRC-passing results, deduplication via existing text-dedup.
- The subcode generator is ~50 LOC (random GF(2) row-reduced basis + coset
  offset application); BP loop and CRC unchanged.
- Start with K=8 sweep {4,8,16,32} on hard-200; budget impact is K× current
  layered-BP elapsed, but layered BP is -16% vs flooding (hb-063) so K=8
  costs ~8× of a very fast baseline.
- Kill-switch: Δrec < +5 at K=8 → close.

**License**: Both papers are academic (published under IEEE open-access or
arXiv preprint). No GPL source involved; implementer reads paper equations only.

**Relation to hb-257**: hb-257 describes "parallel constituent decoders + ML
selection" generically. SCED provides the concrete algorithm with primary-source
backing and measured results. hb-261 supersedes hb-257 in actionability; hb-257
should reference hb-261 as the specific implementation path.

**Draft hb entry**: see §hb-261 below.

### 2. Sync-Score-Gated Unusual-Format Confidence Filter — hb-262, 0.38

**Technique**: Unusual-format messages (NonStandard, /R, contest-format) require
a higher minimum Costas sync_score to pass the decode filter. Standard messages
keep the existing threshold (~3.0). Unusual formats get elevated thresholds
(empirically tuned, e.g. NonStd: 5.0, /R: 4.5).

**Primary source**: WSJT-X Improved v3.1.0 release notes (released 2026-05-22,
https://sourceforge.net/projects/wsjt-x-improved/). The "False Decodes Reduction
(FDR)" feature is described as: "significantly reduces FT8 false decodes without
a reduction of the sensitivity, only increasing required confidence level for
unusual message formats." The description is behavioral (not source-code
derived); no clean-room firewall needed.

**Note on mr-001 coverage**: mr-001 (May 2026) audited WSJT-X Improved through
v3.1.0 and harvested hb-044..hb-048 (DT refinement, passband, two-stage, AP-7).
The FDR mechanism was present in v3.1.0 but was not harvested in mr-001 — the
harvest focused on decoding sensitivity algorithms, not FP-filter extensions.
This is a legitimate gap.

**Measured evidence**: Not quantified in the release notes. However, pancetta's
own fp_format_audit.rs (hb-058) measured 315 /R novels at 0 recovered on
hard-200, and NonStdCall is 7 TP / 11 FP = 39% precision. A score-gated filter
on these formats, preserving high-sync-score instances, should recover precision
at minimal recall cost.

**Applicability**: Very clean.
- Pancetta's CandidateMessage carries sync_score throughout decode; filter is
  ~10-15 LOC in is_plausible() or a new post-decode confidence gate.
- Distinct from hb-058 (binary contest-type reject, GRADUATED): FDR is
  *continuous* — low-confidence unusual formats rejected, high-confidence
  unusual formats pass.
- Distinct from hb-071 (/R pending, binary consideration): FDR naturally
  handles /R via threshold rather than binary rules.

**License**: Behavioral specification from public release notes. No GPL exposure.

**Draft hb entry**: see §hb-262 below.

### 3. Burst-Coherence Hard-Gate for MRC Cross-Cycle Integration — hb-263, 0.28

**Technique**: In pancetta's MRC coherent cross-cycle averaging (hb-075,
GRADUATED), the current soft weighting uses weight_i ∝ |acc_i| (magnitude of
the Costas phase accumulator), linearly downweighting slots with noisy phase
estimates. The proposed refinement adds a *hard exclusion gate*: if |acc_i|
drops below a threshold (proxy for a burst-corrupted or deeply-faded slot),
exclude that slot entirely from the coherent sum rather than linearly
downweighting it.

**Primary source**: arXiv:2604.07004 (April 2026) "Channel Estimation and LDPC
Decoding for Bursty Phase Noise" (Chalmers University of Technology / University
of Toronto). The paper demonstrates 1.4 dB gain from iterative burst-aware
(IBA) LDPC decoding vs. conventional decoding by using Viterbi/BCJR to detect
burst positions and down-weight those positions in the LLR computation. The
burst-detection / hard-exclusion paradigm is the translatable concept; the
specific optical-channel model and BCJR burst detector are not directly
applicable to FT8.

**Translation to FT8**: The MRC Costas rotor magnitude |acc_i| is pancetta's
natural burst-detection oracle:
- |acc_i| ≈ 0 → severe fade or noise burst; current linear weight contributes
  ~0 (soft MRC already handles this case well)
- 0 < |acc_i| << mean → moderate burst; current linear weight still contributes
  some corrupted signal; hard gate would exclude

**Honesty caveat**: pancetta's soft MRC weighting (hb-075) was validated and
graduated cleanly. The added benefit of a hard gate is only present when the
distribution of |acc| values has a significant population of *moderate-burst*
slots (not so faded that they self-suppress, but faded enough that their
contribution adds noise). This requires corpus evidence to justify.

**Measured evidence for FT8**: None yet. This is a diagnostic-first ticket.

**License**: Paper is academic (Chalmers/UofT, no GPL source). The translation
is purely conceptual.

**Draft hb entry**: see §hb-263 below.

## Dead ends and confirmed non-starters

**Bursty Phase Noise LDPC for optical/coherent channels**: arXiv:2604.07004's
Viterbi/BCJR burst detector is designed for coherent optical communication where
phase noise is the dominant impairment. FT8's demodulation is *noncoherent*
(max-magnitude bin selection); phase noise affects coherent paths only. Direct
application would require adopting coherent demodulation first (BICM-ID
iteration, hb-252). Framed conservatively as hb-263 (diagnostic / MRC
refinement) rather than a direct port.

**MSHV SD FT8 Decoder (v2.76.4, Dec 2025)**: MSHV's new "SD FT8 Decoder
(Beta)" derives from WSJT-X algorithms per its own credits. No novel algorithmic
contributions identified beyond the upstream WSJT-X / JTDX lineage already
audited by mr-001/mr-002.

**WSJT-Z (sq9fve)**: Automation fork (Auto-CQ, Auto-Call, band-hopping, DXCC
filtering). No decoder algorithm changes vs WSJT-X baseline. Not a new source.

**GRAND / ORBGRAND**: arXiv:2602.01174 (Feb 2026) "Reducing ORBGRAND Latency
via Partial Gaussian Elimination" reduces ORBGRAND's latency for high-rate short
codes. FT8's (174,91) rate-0.52 code remains far outside GRAND's tractable
guessing range (83 redundancy bits). hb-260 dead-end stands.

**kholia/ft8_lib**: No algorithmic updates in 2025-2026; remains at the same
lightweight LDPC decoder targeted at microcontrollers.
