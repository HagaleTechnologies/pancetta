# AP type 7 (a7) cross-correlation — design spec (hb-048)

**Status:** proposed (Session 1 of 3; design only, no code yet)
**Hypothesis:** hb-048 (mr-001-spawned, 2026-05-24)
**Author:** research harness, 2026-05-31
**Estimated effort:** 3 sessions
**Parent / spawn context:** mr-001 audit flagged "WSJT-X-Improved a7" as
*the* high-leverage idea pancetta's bank didn't have. Source: WSJT-X
mainline commit `f13e31820470291fdd49627287a2dc08f3fa674c` (Joe Taylor,
2021) introducing `lib/ft8_a7.f90`.

## Why this is the next-tier lever after the hb-086 family closes

Pancetta's current AP path (`par_try_ap_decode`,
`pancetta-ft8/src/decoder.rs::3630`) injects known callsigns into the
*current* slot's decode as soft LLR pins (AP1-AP4 + the hb-043
my-call-less paths). It works when the operator already knows who is
called or who is calling — `my_call`, `recent_calls`, `active_qso`.

WSJT-X's a7 inverts the time axis: after slot N decodes callsign C,
build a small set of expected *follow-up* messages from C (roughly 206
templates per decoded call), modulate each one to a complex template,
and *cross-correlate* those templates against slot N+1's complex
baseband at the same audio frequency. Strong correlation (snr7 ≥ 6.0
**and** snr7b ≥ 1.8 second-best margin) means the expected next-slot
message is actually there, even if its sync_score is below the standard
Costas gate.

This is a structurally different mechanism from anything pancetta runs
today:

- **a7 chooses a position by prior expectation** (slot N's decode tells
  us where to listen in slot N+1); the existing AP path chooses positions
  via Costas-pre-gate, then applies AP after the candidate is admitted.
- **a7 does correlation-against-templates**, not LDPC-with-AP-pins, so it
  can rescue signals whose tone constellations are too smeared for LDPC
  to converge even with AP help.
- **a7 needs cross-slot state**, which pancetta's decoder has
  historically not carried.

Diagnostic upper bound is not measured at scope time (see Open
Questions §7). A first-order estimate: any in-progress QSO where slot
N decodes but slot N+1's reply is sub-Costas is a candidate. The
hb-051 ceiling diagnostic (1/8576 with perfect AP info) does **not**
bound a7 — that test was for AP-as-LLR-pins, not template
cross-correlation. a7's value space is fundamentally different.

## Mechanism (~250 words)

**What a7 does in WSJT-X.** After slot N produces a successful decode
with callsign C at frequency f and time t, `ft8_a7_save()` stashes
(C, f, slot_parity) in a small even/odd table. On the next slot of the
same audio-frequency neighborhood (within 2 Hz of f), `ft8_dec7`
generates ~206 candidate next-utterance templates rooted at C:

- Standard message pairs `<C OTHER>` and `<OTHER C>` with each
  callsign already heard at adjacent positions;
- Non-standard wrapped variants `<C> OTHER`;
- CQ variants with grid squares;
- Status messages `RRR`, `RR73`, `73`;
- SNR reports spanning -50 to +50 dB in 1 dB steps.

Each template is source-encoded via the standard FT8 packer →
77-bit payload → LDPC → 174-bit codeword → 8-GFSK tone sequence →
complex baseband `cwave`. The decoder computes
`z = sum(cd(i+lag) · conj(cwave(i)))` over a ±20-sample sliding window
around the previous decode's time offset.

**Bit-position pinning.** a7 templates fix ~28 bits explicitly
(callsign C's bits at either 0-27 or 28-55, plus i3 type bits at 74-76
constraining message structure). The remaining ~146 bits are search:
the template enumeration walks plausible OTHER calls, plausible
reports, and the small structured-message set.

**Acceptance thresholds (WSJT-X starting values).**
`snr7 ≥ 6.0` is the absolute correlation peak; `snr7b ≥ 1.8` is the
ratio between best and second-best lags' correlations. The second test
is the AP-FP filter: random noise can spike one lag but rarely
beats every other lag by 1.8×.

**Previous-sequence filter.** `f0 = -98.0` flags messages already
decoded one slot ago at the same frequency, so the SAME station calling
again doesn't fire a7 against itself uselessly. Saves CPU.

## Architecture fit (~250 words)

**Difference from existing AP1-AP6.** Pancetta's AP path
(`pancetta-ft8/src/ap.rs`,
`pancetta-ft8/src/decoder.rs::par_try_ap_decode`) is per-candidate
LLR injection at *standard-Costas-admitted* candidates. AP1 pins own
call at bits 28-55; AP2 pins a recent caller at bits 0-27; AP3+AP4
pin both during an active QSO; the hb-043 path tries each recent call
at either position when `my_call` is unset. **All variants assume the
candidate was already admitted by Costas sync**, then ask LDPC to
converge with extra constraints. a7 *replaces* the Costas gate with a
template-correlation gate — different decision surface entirely.

**New code needed.**

- `pancetta-ft8/src/a7.rs` (~200 LOC + 10 tests): template generator
  (callsign-bits packer is already in `ap.rs::pack28`; tone-sequence
  encoding from `encoder.rs` is feature-gated behind `transmit`, which
  is fine — a7 will gate on `cfg(feature = "transmit")` for the
  template synthesis side); cross-correlation primitive over complex
  baseband; snr7 / snr7b computation.
- `pancetta-ft8/src/decoder.rs`: new pass
  `a7_cross_correlation_pass(samples, expected_calls)` after the V1
  joint-pair retry but before final emission. New `Ft8Config` fields:
  `a7_enable: bool`, `a7_snr7_threshold: f32` (default 6.0),
  `a7_snr7b_threshold: f32` (default 1.8), `a7_freq_window_hz: f32`
  (default 2.0), `a7_lag_samples: usize` (default 20).
- Per-decoder cross-slot state: `Ft8Decoder` gains
  `prev_slot_calls: Vec<A7ExpectedCall>` (the even/odd table from
  `ft8_a7_save`). Updated at end of each `decode_window_with_ap`;
  consumed at start of next.

**Coordinator hook.** The cleanest fit is to keep cross-slot memory
**inside `Ft8Decoder`** (decoder-owned) rather than threading it
through coordinator. Reason: the decoder already has feedback loops
(multi-pass coherent subtract); adding a one-slot lookback is local
state, not coordinator state. The coordinator already constructs the
decoder once and reuses it across cycles, so the cross-slot table
naturally lives at the right lifetime.

**Integration with `pancetta-qso::autonomous::recently_responded_to`.**
The autonomous operator already maintains a per-callsign HashMap
keyed by `Instant` with a 60s window. We do **not** re-use this for
a7 directly — autonomous's set is "calls I responded to" (a TX-side
defense against spam); a7 needs "calls I decoded last slot at audio
freq f" (an RX-side prior). Same shape, different semantics. The a7
table will mirror the structure (HashMap-ish, time-bounded) but live
in `Ft8Decoder`.

    Each `A7Template` carries `{ message_text, payload_bits: [bool; 77],

  - `pub fn cross_correlate(cd: &[Complex32], cwave: &[Complex32],
    initial_lag: i32, lag_window: i32) -> A7Score` where `A7Score {
    best_lag, snr7, snr7b }`.
  - All gated on `cfg(feature = "transmit")` for the encoder side
    (template synthesis requires the encoder). The `cross_correlate`
    primitive itself is non-transmit (just FFT math).
- 10 unit tests:
  1. `templates_for_known_call_count_206` — verify count matches
     WSJT-X reference within ±5 (we may need fewer SNR steps).
  2. `template_C_at_caller_position_packs_correctly` — round-trip via
     `pack28`.
  3. `template_C_at_called_position_packs_correctly`.

  5. `template_status_RR73_packs_correctly`.
  6. `cross_correlation_matches_self_at_lag_0` — `cd = cwave` → best

  7. `cross_correlation_with_noise_only_fails_threshold` — pure
     white noise → snr7 < 6.0.
  8. `cross_correlation_finds_known_lag_offset` — `cd = shift(cwave, 5)`
     → best_lag = 5, snr7 high.
  9. `cross_correlation_rejects_wrong_call_template` — modulated K1ABC

  10. `prev_slot_filter_dedups_same_call_at_same_freq` — exercise the

### Session 3 — production wiring + eval

- Wire `a7_cross_correlation_pass` into `decode_window_with_ap` after
  the V1 joint-pair-retry pass. Per-slot:
  1. End of slot N: collect successful decodes' (callsign, freq_hz,
     slot_parity) into `Ft8Decoder.prev_slot_calls`. Evict entries
     older than 30 seconds (2 slots).
  2. Start of slot N+1: for each entry in `prev_slot_calls` whose
     `slot_parity != current_parity` (i.e., the previous slot, not
     this one), generate templates and run cross-correlation on the
     residual complex baseband (post-multipass + V1).
  3. Accept decodes where snr7 ≥ threshold AND snr7b ≥ threshold; pass
     through the standard FP funnel (plausibility, continuity filter,
     dedup vs already-decoded set).
- Threshold sweep:
  - snr7 ∈ {5.0, 5.5, 6.0, 6.5, 7.0}
  - snr7b ∈ {1.5, 1.8, 2.2}
  - Per-WAV scorecards on hard-200 + synth + fixtures.
- Cross-slot state plumbing into the research harness: the harness's
  `Ft8Decoder` reuse pattern needs verifying so `prev_slot_calls`
  carries between consecutive WAVs only when consecutive in the
  eval-iteration (it doesn't, because the corpus is shuffled, so the
  eval can only test the within-WAV multipass effect, not the
  across-WAV slot-to-slot effect). **This is the V3-style trap to
  avoid:** measure the right thing. Likely outcome: eval shows weaker
  effect than production will (production has chronological slots);
  document the gap.
- A/B vs `main` on hard-200 (5 runs each, median).

**Session 3 GRADUATE criteria:**
- composite ≥ +0.0005 minimum, +0.001 target.
- hard-200 recovered: +5 minimum, +20 target.
- fixtures: zero regression (binding).
- synth-clean: zero regression (binding).
- elapsed: ≤+10% (binding).
- novel count on hard-200: ≤+30 (FP-filter absorbs most).

## Eval plan

**Headline metric.** hard-200 rec + cross-slot decode count (a new
counter to add: "decodes that came from a7 path" — instrument the new
pass to emit a label).

**Watch.** Novel inflation. WSJT-X's commit log shows multiple later
patches "better suppression of low-confidence false decodes generated
by AP decoding." We expect FP pressure; the strict FP filter
(post-Phase-5) should absorb it, but a regression in novel count would
indicate cross-correlation FPs slipping through.

**Guard (binding):** fixtures + synth-clean must hold; the
callsign-continuity FP filter must hold the precision line.

**Per-WAV scorecard:** instrument with a counter `a7_template_decodes`
so the per-WAV row shows how many of that WAV's recovered decodes came
through a7.

**Estimate.** Realistic:
- hard-200 rec: +5 to +25 (low end if eval-harness shuffle hides the
  cross-slot signal; high end if within-WAV-multipass still benefits
  via templates from already-decoded calls in the same WAV).
- hard-1000 rec: +20 to +50 (more cross-slot opportunities at scale).
- composite: +0.0005 to +0.0020.

## Risk analysis

**Top-3 risks.**

1. **FP pressure (HIGH).** WSJT-X iterated multiple times suppressing
   a7 FPs. The mechanism essentially says "I trust correlation at
   snr7=6 even without Costas-sync admission" — that's a large class
   relaxation. Pancetta's FP filter (strict, post-Phase-5) is the
   designed countermeasure but it's not been stress-tested by an
   AP-FP-generating mechanism since the filter went strict. The
   threshold sweep MUST include the conservative end (snr7 ≥ 7.0,
   snr7b ≥ 2.2) so we know how steep the precision/recall trade is.

2. **State-of-callsigns plumbing in research harness (MEDIUM-HIGH).**
   The eval harness shuffles WAVs and reuses one `Ft8Decoder` across
   the corpus. If `prev_slot_calls` carries between unrelated WAVs,
   we will generate templates for callsigns that don't appear in the
   current WAV — pure CPU waste, possible FPs. Fix in Session 3:
   either (a) reset `prev_slot_calls` per WAV (correct for offline
   eval but undercounts production yield), or (b) tag entries with a
   WAV-id and filter (more realistic). Pick (a) and document the
   under-count; (b) is over-engineering for offline eval.

3. **CPU cost (MEDIUM).** Per-decode in slot N → ~206 templates ×
   (template-synth + cross-correlate). Template synthesis is bounded
   (8-GFSK modulation of a fixed-length payload, ~5ms). Correlation
   is FFT-based, also bounded. Total: ~5ms × 206 × (decodes/slot,
   typically 5-30) = 5-30s per slot at the upper bound. Way over a
   15s wall-clock. **Mitigations**: cap templates per source-decode
   (top-K most common variants only — first cut: 32 templates
   covering caller/called/CQ/RR73/73 + 24 SNR-report variants in 4 dB
   steps); cap source-decodes contributing templates (top-K by SNR,
   K=8); precompute template waveforms once at startup for the small
   non-callsign-dependent suffix set.

**Additional risks (lower).**

- **Bit-exact decode count change.** a7 will add decodes that ft8_lib
  doesn't produce, breaking any fixture that asserts exact decode
  count vs ft8_lib. Audit the fixture set in Session 3 before eval;
  may need to regenerate fixtures or whitelist a7-attributed decodes.
- **Operator's own callsign FP pollution.** a7 templates from
  pancetta's *own* CQ (slot N where we transmitted) could in principle
  feed back as expected-next-slot. Don't include self-call decodes in
  `prev_slot_calls`. Easy guard.

## Open questions / dependencies

1. **Cross-slot vs within-WAV.** a7 needs slot N to feed slot N+1.
   Pancetta's research-harness WAVs are 12.64s windows — they ARE the
   slot. Production sees consecutive slots; eval doesn't. Quantify
   how much of a7's yield is within-WAV (decode A at f, template
   feeds local-pass for OTHER calls in the same WAV) vs cross-WAV
   (production-only). Within-WAV is what eval can measure.

2. **Pancetta-qso QSO state machine hook.** The QSO state machine
   already tracks active QSOs (`pancetta-qso/src/qso_manager.rs`).
   Should a7's template set in slot N+1 *prioritize* templates rooted
   at the active-QSO partner's callsign? **Tentative answer: yes for
   production runtime, no for offline eval.** Production should get
   the priority; offline eval should test the basic mechanism without
   QSO context to keep the test surface clean. Document the
   production-only enrichment as a Session-3 follow-up.

3. **Synergy with hb-087.** hb-087 (callsign-priors-on-residual,
   spec'd 2026-05-31) attacks the same wall with a different lever
   (AP-LLR injection on residual at sub-Costas positions). a7 is
   structurally orthogonal (template correlation vs LDPC-with-AP),
   so they should compose. If both graduate, Session 4-ish work could
   merge prior sources. **Not blocking Session 2 or Session 3.**

4. **Threshold transferability.** WSJT-X's snr7=6, snr7b=1.8 are
   tuned against WSJT-X's correlation power normalization. Pancetta's
   FFT bin scaling may differ. Session 2's decodability micro-test
   should report achieved snr7 / snr7b on a known-good template, and
   Session 3's sweep should bracket the WSJT-X values widely.

5. **Operator gate.** Like the existing AP path, a7 is most valuable
   when operator has an active QSO context. Should it be disabled
   in pure scan-mode? Tentative: enable always, but the templates
   rooted at the operator's callsign get priority weight in
   production (no impact on offline eval since `my_call` is unset
   there). Defer the priority weighting design to Session 3.
