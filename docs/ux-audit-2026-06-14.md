# Pancetta TUI/UX Audit — 2026-06-14

Deep "walk the whole app as a user" audit across 6 flow lanes (input, startup/config/audio,
receive/monitor, DX-hunt/Callers, QSO lifecycle, TX control/freq). Findings below, ranked,
grouped into fix batches. Status: [ ] todo, [x] done.

## Batch 1 — Safety + input correctness (highest operator impact)
Status: ALL items done (2026-06-14). See `fix(tui): UX audit batch 1` commit.
- [x] **TX-policy SAFETY LEAKS**: `p` (TogglePtt) keys the rig with NO policy check — keys carrier
  while DISABLED/after Shift+Q (tui_relay.rs TogglePtt ~921). Shift+T tune also bypasses policy
  (ToggleTune ~836; TuneRequest worker arm tx.rs ~979 has no Disabled hard-mute). FIX: gate PTT
  key-UP and tune on `TxPolicy::allows_any_tx()`; add Disabled hard-mute to the TuneRequest arm.
  DONE: relay refuses the PTT key-UP (state=true) while Disabled (PTT-OFF always allowed) + TUI `p`
  pre-checks `app.tx_policy.allows_any_tx()` for instant feedback; relay refuses tune-START while
  Disabled + TUI Shift+T pre-check; tx.rs TuneRequest arm now hard-mutes on Disabled (catch-all,
  matching TransmitRequest/MultiTransmitRequest).
- [x] **g→DISABLED doesn't abort in-flight TX** (worker only checks policy at dequeue, tx.rs ~334).
  FIX: cycling to Disabled sets the abort_current_tx flag (like Shift+Q). DONE: CycleTxPolicy handler
  sets `abort_current_tx`, stops the CQ loop, and clears the tune tracker when the new policy is Disabled.
- [x] **Two divergent key handlers**: standalone `pancetta-tui` bin uses stale `App::handle_key_event`
  (app.rs ~659: Esc=quit-no-confirm, j/k=scroll, ONLY place 1-5 bound). Production `TuiRunner` is
  authoritative but binds NO digit keys. FIX: make the bin route through TuiRunner (or delete its
  divergent handler), and add 1-5 panel-jump to TuiRunner::handle_key_event. DONE: deleted the
  standalone bin (`src/main.rs` was explicitly "standalone not supported"; `[[bin]]` removed +
  `autobins = false` so a stray main.rs can't resurrect it; unused clap/tracing-* deps dropped),
  deleted `App::handle_key_event`, and added 1-5 panel-jump arms to `TuiRunner::handle_key_event`.
- [x] **No command vs text-input mode**: every letter is both a command and a char; command wins;
  free-text TX unusable; `k`/`s`/`c` fire while "typing" (tui_runner.rs flat match ~578). FIX: explicit
  compose mode (enter via a key, Esc exits; chars→buffer only in compose) so command keys are safe.
  DONE: `/` enters compose (visible `TX> …_` prompt in the status bar); in compose Char/Backspace edit
  the buffer, Enter sends + exits, Esc cancels + exits; OUTSIDE compose the `Char(c)` text catch-all
  was removed so letters are commands only and stray keys never fill the buffer. Callers Enter-reply
  preserved.
- [x] **`r`/`k` fire globally, not gated to QsoStatus focus** (tui_runner.rs ~758) — `k` aborts the
  selected QSO while scrolling Band Activity. FIX: gate r/k to `ActivePanel::QsoStatus`; echo target
  callsign in the abort feedback. DONE: r/k act only when `active_panel == QsoStatus` (else a hint);
  feedback echoes the callsign ("Aborting QSO with W1AW"); QSO-panel control hints note "(this panel
  only)" and help reflects the focus requirement.
- [x] **Cursors not callsign-pinned** → list re-sorts/grows under cursor → Space/Enter/k retarget
  silently. DX Hunter has NO clamp (app.rs ~1418, dx_hunter_scroll positional). Callers clamps the
  override but not the target, and only on Up/Down (app.rs clamp_callers_selection ~1489; not called
  on Home/End/PgUp/PgDn or on decode arrival). FIX: pin both selections to a callsign; re-derive
  index after every list mutation + on decode push. DONE: added `dx_hunter_pinned_call` /
  `callers_pinned_call`; `clamp_dx_hunter_selection` (new) + reworked `clamp_callers_selection`
  re-derive the index from the pinned callsign (fallback clamp if gone); cursor moves repin
  (Up/Down/Home/End/PgUp/PgDn), and decode arrival / spot merge / add_dx_spot re-clamp both. Override
  resets whenever the pinned callsign changes.

## Batch 2 — QSO lifecycle visibility + correctness
Status: ALL items done (2026-06-14). See `fix(qso): UX audit batch 2` commit.
- [x] **Watchdog invisible** (the "infinite loop" complaint): call_count/max/deadline never reach the
  TUI; panel shows only `Exchanges: N` (raw msg-log len, frozen during keep-call since rearm doesn't
  append a Sent msg). FIX: add call_count/manual_call_max/watchdog-deadline to ActiveQsoSnapshotItem;
  render `Call 4/10 · stops 3:12` for manual RespondingToCq/SendingReport.
  DONE: added `call_count`/`max_calls`/`watchdog_deadline` to `ActiveQsoSnapshotItem`, populated in
  `snapshot_item_from_progress` from metadata + `TimeoutConfig` ONLY for manual RespondingToCq/
  SendingReport (zero/None otherwise); mirrored onto `ActiveQsoBanner`/`QsoStatus` + the relay map;
  `qso_status.rs` renders `Call N/M · stops M:SS` (live countdown each frame) in the single-detail
  panel AND the multi-QSO table. `rearm_manual_calls_at` now appends a Sent record so the activity
  counter no longer freezes during keep-calling.
- [x] **last_tx_text empty for engine TX** (raw_text=String::new() at respond_to_cq_with ~554,
  respond_to_caller ~757, auto-reply ~1083; rearm appends nothing). Panel "TX:" line blank. FIX:
  populate raw_text via MessageExchange::generate_message at creation; append on rearm.
  DONE: new `QsoManager::render_sent_text` renders each Sent `QsoMessage.raw_text` via
  `MessageExchange::generate_message` at all four sites (respond_to_cq_with, respond_to_caller, the
  auto-reply in process_message_for_qso, and rearm_manual_calls_at which now appends a Sent record).
  resend_last_tx still re-emits the last Sent MessageType.
- [x] **Failure reason never surfaced** (Superseded/Timeout/sender-mismatch all just vanish from the
  snapshot; security-mismatch is log-only). FIX: emit a StatusUpdate carrying QsoFailureReason on
  terminal transitions; brief toast.
  DONE: the coordinator QsoEvent loop's StateChanged arm detects a transition INTO Failed and emits a
  `MessageType::StatusUpdate` via the existing QSO→TUI status channel ("QSO with W1AW failed: watchdog
  timeout"), mapped by new `failure_reason_text`. Sender-mismatch discards stay log-only (they never go
  terminal — nothing to surface).
- [x] **Completed never shown** (terminal QSOs filtered from snapshot; ladder_view None). FIX: emit a
  "QSO with X logged" status on QsoCompleted (and/or keep terminal in snapshot ~1 slot with a badge).
  DONE: the QsoCompleted arm emits "QSO with W1AW logged (RST -12/+03)" via the same status channel.
  (Keep-terminal-visible-1-slot was the optional nice-to-have — DEFERRED; the status line is the
  must-have and is in.)
- [x] **Multi-QSO snapshot order non-deterministic** (HashMap.values() unsorted, build_active_qso_snapshot
  ~795) → cursor jumps between QSOs → `k` aborts wrong one. FIX: sort by (start_time, callsign); key
  the TUI qso_cursor on qso_id not position.
  DONE: `build_active_qso_snapshot` collects to a Vec and `sort_by((start_time, callsign))`; the TUI
  pins selection on `qso_pinned_id` and re-derives `qso_cursor` each snapshot (`clamp_qso_selection`),
  cursor moves re-pin (same pattern as Batch 1's DX/Callers cursors).
- [x] **CQer completion path unwired**: CallingCq→WaitingForReport, then NO arm out of WaitingForReport
  (determine_state_transition ~1141) → we-CQed QSOs never complete/log, time out. FIX: add CQer forward
  arms; make ladder_view role-aware (CQer WaitingForReport currently blanks).
  DONE: added `WaitingForReport + ReportAck → WaitingForConfirmation` (sender-verified) + the matching
  relevance arm + the `WaitingForReport + ReportAck → FinalConfirmation` reply in exchange.rs; the
  existing `WaitingForConfirmation + 73/RR73 → Completed` closes it. Ladder is now ROLE-aware: added
  `QsoRole::{Cqer,Caller}` to `QsoMetadata` (Cqer latched in start_cq, Caller elsewhere);
  `ladder_view(QsoRole)` shows CQer rungs for the shared middle states. CQer `their_callsign` latched on
  first advance so the log carries the right call. Full we-CQed sequence→Completed test added. Only the
  state machine + ladder are wired — autonomous CQ-answering stays Phase-5-gated (the Manual-only
  auto-reply gate is unchanged). DEFERRAL: operator `c` still drives the repeating-CQ *text* path
  (tx.rs/tui_relay.rs) which never creates a CallingCq QSO object, so end-to-end manual-CQ completion
  needs that separate wiring (out of Batch 2's allowed-files scope); the completion arms are ready for it.
- [x] **supersede key case-sensitive** (qsos_by_callsign verbatim) — case mismatch could re-spawn dupes.
  FIX: normalize uppercase in add_callsign_mapping/supersede.
  DONE: callsign map keyed uppercase in add/remove_callsign_mapping, supersede_active_qsos_for, and
  check_duplicate. Case-insensitive supersede test added.
- [x] **ADIF grid dropped on common close** (SendingReport-close hard-codes grid_square:None ~1230) —
  decoded DX grid never logged. FIX: latch their grid from opening CqResponse → Completed metadata.
  DONE: process_message_for_qso latches any incoming CqResponse grid into `metadata.grids.theirs`; the
  CQer path additionally threads it via a new `WaitingForReport.their_grid` field → WaitingForConfirmation
  → Completed. Grid-into-Completed tests added for both roles.

## Batch 3 — Config + display polish
Status: ALL items done (2026-06-14). See `fix(ux): UX audit batch 3` commit.
- [x] **Partial pancetta.toml silently reverts ALL config to defaults** (only autonomous/metadata have
  serde(default); skipped-source logged at debug only) → comes up N0CALL/default audio; also breaks the
  device-picker persistence (writes [audio]-only file that then fails to parse). FIX: `#[serde(default)]`
  on the 5 top-level Config sections; warn + TUI-surface on parse-skip; device-picker writes full config.
  DONE: added `#[serde(default)]` on all 5 top-level Config fields AND as a container attr on each
  section struct (StationConfig/AudioConfig/RigConfig/UiConfig/NetworkConfig) so even a present-but-partial
  section fills missing fields from defaults (the operator's real trap). Loader now `warn!`s + records a
  human-readable warning on a parse/validation skip (FileNotFound stays quiet); `Config::load_default_with_warnings`
  + `ConfigLoader::load_warnings()` surface them; main.rs prints them at startup AND the coordinator pushes
  each as a `TuiMessage::Error{component:"config"}` banner. The serde-default fix makes the device-picker's
  `[audio]`-only file load cleanly (no full-config rewrite needed). Tests: partial-[station], [audio]-only
  deserialize, and an [audio]-only persist→load round-trip.
- [x] **Speakers-misconfig warning only in log, not TUI** (stream.rs ~483) — the exact bug the operator
  hit. FIX: route to TUI via MessageType::Error + a persistent "TX→system default" badge.
  DONE: `AudioStreamManager`/`AudioManager` now expose `output_is_system_default()`; the audio thread, on
  start, latches a coordinator `Arc<AtomicBool>` and emits a `MessageType::Error` describing the misconfig;
  the TUI relay pushes `AudioOutputDefault{is_default}` and the station panel renders a persistent
  "⚠ TX→system default" badge on the Device line.
- [x] **`--list-audio` referenced in messages but doesn't exist**; `test-audio --list` is a stub. FIX:
  implement device enumeration; correct the message text.
  DONE: `test-audio --list` now enumerates via `pancetta_audio::device::list_input_devices()/list_output_devices()`,
  flagging the system default; stub-on-no-flag points at the real command. Fixed a latent clap `-d` collision
  (device/duration/global-debug) by making `--device`/`--duration` long-only. stream.rs + info_command text
  corrected to `pancetta test-audio --list`. CLI test asserts success + headers.
- [x] **No rig-connected indicator** (connect failure logs error! + continues; TUI shows "---" S-meter).
  FIX: rig status badge (Connected/No rig/Polling-failed) from the hamlib poll loop.
  DONE: new `RigConnState` (NotConnected/Connected/PollingFailed) in a coordinator `AtomicU8`, written by the
  hamlib connect path (real rig only) and the poll loop (PollingFailed after the crash-warn threshold,
  recovers to Connected); TUI relay pushes `RigStatusUpdate`; station panel renders "RIG: ✓ connected" /
  "RIG: ⚠ polling failed" / "RIG: ✗ not connected". PTT/keying untouched.
- [x] **Waterfall TX cursor shows manual offset, not live TX freq** (ui/mod.rs ~447 reads
  tx_frequency_offset; should use tx_now_sending.freq_hz when sending) — the 1350→2300 visual bug.
  DONE: `render_waterfall` uses `tx_now_sending.freq_hz` (labeled "… (LIVE)") when sending, falling back to
  the manual `tx_frequency_offset` only when idle. The widget green cursor/label follow the passed offset.
  Test asserts the App-state branch the renderer reads.
- [x] **DF/DT columns clipped** (band_activity.rs DF Length(4) for up to "+2800"; DT Length(4) for
  "-15.2"). FIX: widen DF≥6, DT≥5.
  DONE: DT→Length(5), DF→Length(6). Also: DX Hunter SNR for a pure network spot with no SNR now renders
  "---" (extracted `format_dx_snr`, unit-tested) instead of "+0".
- [x] **TX offset clamp 100–3000 wider than modulator 200–2500** (tui_runner.rs ~647); multi-TX goes
  negative below 200. FIX: clamp to 200–2500.
  DONE: all four arrow / `[` / `]` clamp sites now `.max(200.0)` / `.min(2500.0)`. Test hammers `[`/`]` and
  asserts the 200/2500 bounds.
- [x] **Late-TX 30s defer invisible** (looks dead); pre-PTT F8 abort leaves QUEUED strip stale; multi-TX
  mislabeled "1 NOW + N-1 QUEUED" though concurrent. FIX: surface deferred/eta in TxItem; clear strip on
  pre-PTT abort; label multi-TX as concurrent.
  DONE: (a) `TxSchedule.deferred` + a `deferred` flag threaded through `TxItem`/`TxQueueItem`; on a defer the
  single-TX arm re-emits the QUEUED item with `deferred:true` and the strip shows "→ deferred 30s". (b) the
  pre-PTT F8 abort now sends `send_tx_queue_status(None, &[])` before `continue` (the abort fires before the
  TxStatusGuard exists). (c) the strip treats non-deferred queued items sharing the on-air slot as
  concurrent and renders "▶ NOW ×N: a | b | …".
- [x] **TX-policy banner never seeded** (TxPolicyStatus defined+consumed but never produced). FIX: emit
  one authoritative TxPolicyStatus at TUI start (or remove the dead variant).
  DONE: the coordinator command loop seeds one `TuiMessage::TxPolicyUpdate` from the real `tx_policy` atomic

  its still-correct relay consumer were left in place — harmless; the seed uses the live TxPolicyUpdate path.)
